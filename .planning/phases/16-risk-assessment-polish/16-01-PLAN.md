<objective>
Add a minimal “review” experience for questionnaires:
- CRO/Risk Manager can compare answers vs the last submitted cycle (only show previous answers when changed).
- CRO/Risk Manager can request clarification per section, and Risk Owner can respond with a single reply (no threaded comments).
- New questionnaire sends use a v2 template that adds two quantified questions (likelihood + worst-case severity) while keeping v1 history readable.
</objective>

<tasks>

<task type="auto">
  <name>Task 1: Lock the template contract (audit v1 + introduce v2)</name>
  <action>
1) Audit the current v1 contract and make it consistent across FE/BE:
- Frontend v1 currently renders `q10_support_needed` but backend `QUESTION_KEYS_V1` does not list it.
- Fix by aligning backend’s v1 keys list with the frontend question set (do not remove q10 from FE in this phase).

2) Introduce template version `v2` for new sends:
- Keep `template_key = "risk_owner_reassessment"`.
- New template version string: `v2`.
- v2 adds two new quantified answers (stored as JSON numbers):
  - `risk_assessment.q11_likelihood_12m` (required, integer 1–5)
  - `risk_assessment.q12_worst_case_impact` (required, integer 1–5)

3) Define and document “previous cycle” lookup (used by compare and summary):
- “Previous cycle” = previous **submitted** questionnaire for the same risk:
  - the row with the greatest `submitted_at` that is strictly **before** the current questionnaire’s `submitted_at`
  - if the current questionnaire is not submitted, use the greatest `submitted_at` overall (latest submitted)
- If no previous submitted questionnaire exists, compare UI must gracefully show “no previous cycle” (no placeholders everywhere).
  </action>
  <done>Backend and frontend agree on v1 keys, v2 keys, and the “previous cycle” definition is explicitly documented and used consistently.</done>
</task>

<task type="auto">
  <name>Task 2: Backend — version-aware validation + sending v2</name>
  <action>
Update `backend/app/services/risk_questionnaire_service.py` and `backend/app/api/v1/endpoints/risk_questionnaires.py` so:

1) Sending:
- `/risks/{risk_id}/questionnaires/send` uses `QUESTIONNAIRE_TEMPLATE_VERSION = "v2"` for newly created questionnaires.
- No DB migration is required for template version change (it is already stored per-row).

2) Submission validation:
- Replace the hardcoded `validate_submit_answers_v1(...)` usage with a version-aware validator:
  - `v1`: required keys remain the existing required set (plus the v1 key audit fix from Task 1).
  - `v2`: required keys = v1 required keys + `q11_likelihood_12m` + `q12_worst_case_impact`.
- Add minimal type/range validation for v2 quantified questions:
  - `q11_likelihood_12m` and `q12_worst_case_impact` must be ints 1–5.
  - Return a clear 400 with which keys are missing/invalid (keep structure consistent with existing “missing required answers” error).

3) Document the contract:
- Add docstrings/comments near the constants so future changes do not reintroduce key drift between FE/BE.
  </action>
  <done>New questionnaires are created as v2; v1/v2 submissions validate correctly and return consistent 400 errors when invalid.</done>
</task>

<task type="auto">
  <name>Task 3: Backend — expose previous submitted answers (opt-in)</name>
  <action>
Extend `GET /questionnaires/{questionnaire_id}` to support compare UI without extra client-side joins:

- Add a query param: `include_previous: bool = false`.
- If `include_previous=true`, include a new optional field in the response, e.g.:
  - `previous_submission: { id, submitted_at, template_version, answers } | null`
- “Previous submission” must follow the definition from Task 1.
- Keep payload minimal: include only what is needed for diffing (id + submitted_at + answers + version).

Concrete contract (example JSON shape):
- Existing fields remain unchanged.
- New field added at top-level:
  - `previous_submission`:
    - `null` if no previous submitted questionnaire exists
    - otherwise:
      - `id: number`
      - `submitted_at: string (ISO datetime)`
      - `template_version: string`
      - `answers: object` (same shape as `answers` on the current questionnaire)

Implementation notes:
- Update `backend/app/schemas/risk_questionnaire.py` to add a small nested schema for `previous_submission`.
- Ensure department scoping/permissions remain identical to the current `GET /questionnaires/{id}` behavior.
  </action>
  <done>Frontend can fetch a questionnaire plus its previous submission in one request when `include_previous=true`.</done>
</task>

<task type="auto">
  <name>Task 5: Frontend — compare mode (“Compare to last cycle”)</name>
  <action>
Add a compare mode to `RiskQuestionnaireDetail`:
- Add a “Compare to last cycle” toggle.
  - Default ON for CRO/Risk Manager and Risk Owner (per Phase 16 context).
  - Allow users to toggle it OFF if they want a cleaner view.
- When compare mode is ON, load questionnaire with `include_previous=true`.
- Diff logic (must be deterministic and documented):
  - Only compare keys that exist in the active template’s question set.
  - Normalize strings by trimming whitespace before comparing.
  - Treat `null/undefined/""` as “missing”.
- Rendering:
  - Always show the current answer in the normal field.
  - If the previous answer exists and is different, show previous value beneath in muted text + a subtle “Changed” badge.
  - If no previous submission exists, do not render “Changed” everywhere; show a single small note like “No previous cycle available.”
  </action>
  <done>Reviewers can scan only the changed answers quickly; unchanged fields remain uncluttered.</done>
</task>

<task type="auto">
  <name>Task 6: Clarification requests (per section, single reply)</name>
  <action>
Implement a minimal clarification workflow without per-question threads:

Backend:
- Add a new model/table `risk_questionnaire_clarifications` (Alembic migration required) with fields:
  - `id`
  - `questionnaire_id` (FK)
  - `section_key` (string; matches FE section titleKey)
  - `question_keys` (JSON list of question keys; optional)
  - `request_message` (text)
  - `requested_by_user_id` (FK)
  - `requested_at` (timestamp)
  - `response_message` (text; nullable)
  - `responded_by_user_id` (FK; nullable)
  - `responded_at` (timestamp; nullable)
- Add API endpoints:
  - `POST /questionnaires/{id}/clarifications` (CRO/Risk Manager only) to create a request
    - payload:
      - `section_key: string`
      - `request_message: string`
      - `question_keys?: string[]`
  - `GET /questionnaires/{id}/clarifications` (any user who can read the questionnaire) to list
  - `POST /questionnaires/{id}/clarifications/{clarification_id}/respond` (Risk Owner only) to add a single response
    - payload:
      - `response_message: string`
- Response shape should include enough to render without extra joins:
  - requester/responder name/id, requested_at/responded_at, messages, section_key, question_keys

- Notifications:
  - Add `NotificationType.QUESTIONNAIRE_CLARIFICATION_REQUESTED` and localized EN/CS messages.
    - Note: `Notification.type` is backed by an enum + DB constraint; add/adjust an Alembic migration if required (follow the enum/constraint pattern used in Plan 14-03).
  - Notify the assignee (Risk Owner) when a clarification is requested.
  - (Optional) Also notify the requester when a response is submitted; keep optional unless it is trivial to add.

Frontend:
- In `RiskQuestionnaireDetail`, in each section header:
  - If current user is CRO/Risk Manager and questionnaire is submitted: show “Request clarification”.
  - The dialog collects `request_message` and optional checkboxes for questions in that section.
- Display clarifications inline per section:
  - Show open requests (message + requested_at + requester name).
  - If current user is Risk Owner: show a “Respond” textarea + submit button (one reply).
  - After responding, show the response message and responded_at.

Non-goals:
- No per-question comment threads, no reactions, no multi-message chat, no reopening submitted questionnaires.
  </action>
  <verify>cd backend && pytest -q</verify>
  <done>Clarifications can be requested per section and responded to once, are persisted, and generate the expected notification(s).</done>
</task>

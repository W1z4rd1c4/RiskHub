# Plan 152-04: KRI Deletion Should Archive, Not Hard Delete

## Priority: 🟠 HIGH (audit compliance)

## Goal
Make KRI “deletion” behave like Risk/Control deletion: **archive/soft-delete** so:
- KRI records remain for auditability
- `KRIValueHistory` is preserved (no cascade data loss)
- KRI no longer appears in normal operational views/metrics

## Current Issue (Confirmed)
Approval execution hard-deletes KRIs:
- `backend/app/api/v1/endpoints/approvals.py` (KRI DELETE path uses `await db.delete(kri)`)

Note: `DELETE /kris/{id}` also hard-deletes KRIs for privileged users today, which is inconsistent with “archive” semantics.

## Design Decision
Use **soft-delete via explicit archive fields** on `KeyRiskIndicator`:
- `is_archived: bool` (default `False`)
- `archived_at: datetime | None`
- `archived_by_id: int | None` (FK to `users.id`)

Rationale:
- Minimal change (no enum/status refactor)
- Explicit audit fields independent from Activity Log
- Avoids cascade delete of history rows

## Implementation Steps

### 1) Model + Migration
- [MODIFY] `backend/app/models/key_risk_indicator.py`
  - Add `is_archived`, `archived_at`, `archived_by_id` (+ relationship to `User` if needed for display/audit).
- [NEW] Alembic migration
  - Add columns to `key_risk_indicators`
  - Set defaults (`is_archived` default false)
  - Add index on `is_archived` (frequent filter)

### 2) Centralize “active KRI” filter
Add a small helper (location flexible; pick one and use everywhere):
- `backend/app/core/` helper like `kri_is_active_clause()` or a local constant used in endpoints/services.

Definition:
- Active KRI = `is_archived == False`

### 3) Change delete semantics (approval + direct endpoint)
- [MODIFY] `backend/app/api/v1/endpoints/approvals.py`
  - When executing `ApprovalActionType.DELETE` for `ApprovalResourceType.KRI`, **set archive fields** instead of `db.delete`.
  - Also emit a KRI entity Activity Log entry (`ActivityAction.ARCHIVE`) to match Risk/Control audit behavior.
- [MODIFY] `backend/app/api/v1/endpoints/kris.py`
  - Privileged `DELETE /kris/{id}` should archive (set the same fields) and log `ActivityAction.ARCHIVE`.
  - Block mutation/recording on archived KRIs:
    - `POST /kris/{id}/values` should 404/409 if archived (unless you intentionally allow privileged backfills; decide and document).
    - `PUT /kris/{id}` should reject archived KRIs (unless `include_archived` + privileged).

### 4) Ensure archived KRIs are excluded everywhere by default
Update all KRI-facing queries so archived items do not leak into business views/metrics:
- [MODIFY] `backend/app/api/v1/endpoints/kris.py`
  - `GET /kris` list
  - `GET /kris/{id}` (return 404 unless `include_archived=true` for privileged/admin)
  - `GET /kris/breaches`, `/overdue`, `/due-soon`, history endpoints where appropriate
- [MODIFY] `backend/app/services/kri_deadline_service.py`
  - Skip archived KRIs in notification generation
- [MODIFY] `backend/app/services/kri_history_service.py`
  - Exclude archived KRIs from `get_overdue_kris` / `get_due_soon_kris`
- [MODIFY] `backend/app/api/v1/endpoints/dashboard.py`
  - Any `KeyRiskIndicator` aggregate queries should exclude archived KRIs (breach counts, KRI health %, overdue KRI counts, risks-with-kri, etc.)

Optional (recommended):
- Add `include_archived: bool = False` query param to list endpoints, restricted to privileged/admin roles, for audit review.

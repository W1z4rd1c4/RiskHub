# Plan 152-03: Investigate & Fix KRI Period Semantics Bug

## Priority: 🔴 CRITICAL (data integrity)

## Problem
KRI period semantics may allow non-privileged submissions to target current/future period ends, which could incorrectly mark prior closed periods as "reported" and skew overdue/due logic.

**Suspected locations:**
- `kris.py:571` - Non-privileged submissions can target current period end
- `kri_history_service.py:165` - `record_value()` updates `last_period_end`
- `kri_history_service.py:220` - Period end handling

**Impact:** History records may have incorrect period associations, breaking audit trail and deadline calculations.

---

## Investigation Required

### Step 1: Analyze Current Period Logic
View and understand:
1. `kris.py` - KRI value submission endpoint
2. `kri_history_service.py` - How `record_value()` handles `period_end`
3. `kri_deadline_service.py` - How deadlines are calculated

### Step 2: Identify Edge Cases
- What happens when user submits for "current" period?
- What happens when period_end is in the future?
- How does `last_period_end` update logic work?

### Step 3: Determine Fix
Based on findings, implement one of:
- Validate `period_end` is <= today
- Prevent re-reporting for already-closed periods
- Add period validation before updating `last_period_end`

---

## Proposed Changes

### [MODIFY] [kris.py](../../../backend/app/api/v1/endpoints/kris.py)

Add period_end validation in submission endpoint:
```python
# Before recording value:
if period_end and period_end > date.today():
    raise HTTPException(
        status_code=400,
        detail="Cannot submit KRI value for future period"
    )
```

### [MODIFY] [kri_history_service.py](../../../backend/app/services/kri_history_service.py)

Add guard in `record_value()`:
```python
# Only update last_period_end if it's newer than current
if not kri.last_period_end or period_end > kri.last_period_end:
    kri.last_period_end = period_end
```

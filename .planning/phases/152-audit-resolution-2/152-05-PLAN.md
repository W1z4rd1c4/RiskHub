# Plan 152-05: Entity-Level Activity Logs on Approval Execution

## Priority: đźź  HIGH (audit completeness)

## Goal
When an approval is **approved** and the action is **executed**, write **entity-level** activity log entries (not just approval-level entries) so audit trails answer:
- Who actually changed the Risk/Control/KRI?
- What changed (fields + old/new)?
- When it changed?
- Which approval caused the change?

## Current Issue (Confirmed)
`POST /api/v1/approvals/{id}/approve` logs only an approval-level entry:
- `backend/app/api/v1/endpoints/approvals.py` logs `ActivityEntityType.APPROVAL` + `ActivityAction.APPROVE`
- It does **not** log `RISK` / `CONTROL` / `KRI` entity changes that occur during execution

## Design Decisions

### 1) â€śDELETEâ€ť approvals map to **ARCHIVE** actions
The app archives these entities (not hard-deletes them), so activity logs must use:
- `ActivityAction.ARCHIVE` for Risk/Control/KRI â€śDELETEâ€ť approvals

### 2) Log only applied changes
For EDIT approvals, log the subset of fields that were actually applied (`hasattr(...)`), so the audit trail matches reality.

### 3) KRI value operations log `KRI_VALUE` too
Approval execution can create/update KRI history:
- Value submission approval â†’ creates a `KRIValueHistory` row
- History correction approval â†’ updates a `KRIValueHistory` row (and may update KRI current fields)

For audit completeness, log:
- `ActivityEntityType.KRI_VALUE` with `ActivityAction.CREATE` / `UPDATE`
- and `ActivityEntityType.KRI` updates when `current_value/last_period_end/last_reported_at` change

### 4) Normalize change values to JSON-safe types
Activity log `changes` must be JSON-serializable. Normalize `date`/`datetime` to `.isoformat()` before persisting.

## Implementation Steps

### 1) Add entity-level logs during approval execution
- [MODIFY] `backend/app/api/v1/endpoints/approvals.py`
  - In `approve_request()`, inside the `should_apply_changes` block:
    - On `ApprovalActionType.DELETE`:
      - Risk â†’ set `status=archived`, log `RISK` + `ARCHIVE` + `description="...approval #..."`
      - Control â†’ set `status=archived`, set `updated_by_id`, log `CONTROL` + `ARCHIVE`
      - KRI â†’ set archive fields (`is_archived`, `archived_at`, `archived_by_id`), log `KRI` + `ARCHIVE`
    - On `ApprovalActionType.EDIT`:
      - Risk/Control â†’ apply pending changes, log entity `UPDATE` with `changes=<applied subset>`
      - KRI:
        - History correction approval â†’ apply correction, log `KRI_VALUE` `UPDATE`, log `KRI` `UPDATE` if current fields changed
        - Value submission approval â†’ record history, log `KRI_VALUE` `CREATE`, log `KRI` `UPDATE` if current fields changed
        - Generic edit (+ optional value record) â†’ log `KRI_VALUE` `CREATE` if value recorded, and `KRI` `UPDATE` with applied changes

### 2) Normalize `changes` in the logging helper
- [MODIFY] `backend/app/core/activity_logger.py`
  - Ensure `log_activity(..., changes=...)` normalizes:
    - `datetime` / `date` â†’ `.isoformat()`

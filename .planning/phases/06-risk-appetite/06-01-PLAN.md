# Plan: 06-01 Risk Appetite Backend

## Objective
Create separate `KeyRiskIndicator` table with numeric limits, linked to Risks. Replace existing text-based KRI fields.

## Design Decisions
- ✅ Separate KRI table (multiple KRIs per Risk)
- ✅ Numeric limits (`lower_limit`, `upper_limit`) for automatic breach detection
- ✅ Each KRI **must** link to a Risk (`risk_id` NOT NULL)
- ✅ A Risk can have zero or many KRIs
- ❌ No process fields (use existing `Risk.department_id`)
- ❌ Deprecate old `Risk.kri_*` text fields

## Tasks

### 1. Create KRI Model
`backend/app/models/key_risk_indicator.py`:
```python
class KeyRiskIndicator(Base):
    __tablename__ = "key_risk_indicators"
    id: Mapped[int] = mapped_column(primary_key=True)
    
    # Required FK to Risk
    risk_id: Mapped[int] = mapped_column(ForeignKey("risks.id"), index=True)
    
    # Core KRI fields
    metric_name: Mapped[str] = mapped_column(String(500))
    current_value: Mapped[float]
    lower_limit: Mapped[float]
    upper_limit: Mapped[float]
    unit: Mapped[str] = mapped_column(String(50), default="%")
    
    # Timestamps
    last_updated: Mapped[datetime] = mapped_column(default=func.now())
    
    # Relationships
    risk: Mapped["Risk"] = relationship("Risk", back_populates="kris")
```

### 2. Update Risk Model

### 4. Pydantic Schemas
`backend/app/schemas/kri.py`:
- Computed `breach_status`: `above` if value > upper, `below` if value < lower, else `within`

### 5. API Endpoints
`/api/v1/kris`:
- GET (list, filter by risk_id or breach_status)
- POST (create, requires risk_id)
- PUT/{id} (update value/limits)
- DELETE/{id}
- GET /breaches (only breached)

### 6. Update Risk API
Include `kris` list in `GET /risks/{id}` response.

<objective>
Create a reusable production script foundation with strict preflight validation before any container install/upgrade action.
</objective>

<tasks>

<task type="auto">
  <name>Task 1: Build shared script library for idempotent Docker operations</name>
  <files>scripts/prod/lib/common.sh</files>
  <action>
Implement shared helpers used by all install/ops scripts:
- strict bash mode (`set -euo pipefail`),
- structured logging and error handling,
- deterministic container naming and image-tag helpers,
- idempotent create/replace helpers for networks and containers.
- consistent CLI flag parsing across scripts:
  - `--backend-env`, `--frontend-env`
  - `--dry-run`, `--yes`, `--verbose`
- redaction helpers: never echo secrets (`DATABASE_URL`, `SECRET_KEY`, `REDIS_PASSWORD`, `WEBHOOK_SECRET`)

Keep functions side-effect-free unless explicitly named as mutating operations.
  </action>
  <verify>All prod scripts source one shared helper library; duplicated shell logic is minimized.</verify>
  <done>Script behaviors are consistent across deploy/upgrade/rollback paths.</done>
</task>

<task type="auto">
  <name>Task 2: Implement production preflight validation</name>
  <files>scripts/prod/lib/preflight.sh, scripts/prod/preflight.sh</files>
  <action>
Implement preflight checks for:
- required binaries (`docker`, `bash`, `curl`),
- docker daemon reachable (user can run `docker ps`),
- required env files and required keys,
- production guardrails mirroring backend startup expectations,
- external PostgreSQL URL sanity:
  - must not be the default placeholder
  - fail if hostname is `db` (compose smell)
- backend/frontend port collision checks on host.

Preflight must fail fast with actionable remediation text.

Add optional DB connectivity check:
- `--check-db` flag (default on for `deploy.sh`, optional off for `--dry-run`)
- implemented via ephemeral backend container:
  - `docker run --rm --env-file backend.env <backend-image> python -c "... SELECT 1 ..."`
- this check must not print the full `DATABASE_URL`.
  </action>
  <verify>Invalid/missing config causes clear non-zero exit before any container mutation occurs.</verify>
  <done>Operators can safely validate host + config readiness before installation.</done>
</task>

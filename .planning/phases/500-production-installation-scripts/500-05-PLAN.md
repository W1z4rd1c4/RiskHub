<objective>
Provide production lifecycle orchestration scripts for deploy, upgrade, rollback, and operations visibility across independent backend/frontend containers.
</objective>

<tasks>

<task type="auto">
  <name>Task 1: Implement primary deploy orchestration script</name>
  <files>scripts/prod/deploy.sh</files>
  <action>
Implement deterministic order:
1) preflight (including optional DB check),
2) install redis,
3) build backend image tag (no container swap yet),
4) run migrations,
5) run DB bootstrap (RBAC + departments + initial admin),
6) install backend API container,
7) install backend scheduler container (recommended default on),
8) build frontend image tag,
9) install frontend,
10) smoke test.

Support non-interactive mode for automation and an explicit confirmation gate for production runs.
  </action>
  <verify>Single command performs full deployment without compose db dependencies.</verify>
  <done>Operators can deploy complete split-container stack with one orchestrated script.</done>
</task>

<task type="auto">
  <name>Task 2: Implement upgrade and rollback scripts with image tag control</name>
  <files>scripts/prod/upgrade.sh, scripts/prod/rollback.sh</files>
  <action>
Add scripts to:
- upgrade backend/frontend to target image tags,
- capture previous running image tags before replacement,
- rollback backend/frontend independently or together,
- preserve env/config files across rollbacks.

Implementation guidance:
- avoid state files; record previous image reference via docker labels on the *new* container (for example `com.riskhub.previous_image=<old image>`).
- upgrades must run migrations and bootstrap before swapping the backend API container.
- rollback must require an explicit flag like `--i-understand-db-wont-downgrade` and must not run any DB downgrade.
  </action>
  <verify>Rollback script restores known previous container image tags after a failed upgrade drill.</verify>
  <done>Deployment lifecycle includes practical recovery path.</done>
</task>

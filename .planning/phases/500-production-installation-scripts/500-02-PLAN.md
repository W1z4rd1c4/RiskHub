<objective>
Implement backend installation and migration scripts that run against external PostgreSQL without relying on a local DB container.
</objective>

<tasks>

<task type="auto">
  <name>Task 1: Implement Redis container install/replace script</name>
  <files>scripts/prod/install_redis.sh</files>
  <action>
Create `install_redis.sh` to run a production Redis container:
- container name `riskhub-redis`
- docker network `riskhub-network` with alias `redis`
- named volume `riskhub-redis-data:/data`
- require password via `REDIS_PASSWORD` from backend env (`redis-server --appendonly yes --requirepass ...`)
- restart policy `unless-stopped`

Do not start PostgreSQL in any form.
  </action>
  <verify>Running script twice results in a single redis container with the expected volume and network alias.</verify>
  <done>Production Redis is installed and reachable for backend startup guards.</done>
</task>

<task type="auto">
  <name>Task 2: Implement backend container install/replace script (API + scheduler instances)</name>
  <files>scripts/prod/install_backend.sh</files>
  <action>
Create `install_backend.sh` to:
- load backend env file,
- run preflight checks,
- build or pull the backend image,
- create/replace backend container idempotently,
- attach to configured Docker network,
- expose no host ports by default (frontend-only access),
- configure restart policy and log path mounts.

Do not invoke `docker compose` or any `db` service for this path.

Add `--instance api|scheduler`:
- `api` instance:
  - container name `riskhub-backend`
  - network alias `backend`
  - `ENABLE_SCHEDULER=false`
  - default workers 4 (overrideable via flag)
  - optional `--publish-backend=127.0.0.1:8000:8000` for ops/debug only
- `scheduler` instance:
  - container name `riskhub-backend-scheduler`
  - no network alias required
  - `ENABLE_SCHEDULER=true`
  - workers MUST be forced to 1 (single-process scheduler safety)
  </action>
  <verify>Running script twice results in a single healthy backend container at the expected image tag.</verify>
  <done>Backend can be installed/upgraded independently with external DB connectivity.</done>
</task>

<task type="auto">
  <name>Task 3: Implement external PostgreSQL migration runner</name>
  <files>scripts/prod/run_migrations.sh</files>
  <action>
Create migration script that:
- uses the backend image and backend env file,
- runs `alembic upgrade head` in an ephemeral container,
- fails fast on migration errors,
- logs migration output for auditability.

Require explicit confirmation or `--yes` for production migration execution.

Print a one-line reminder that DB rollbacks are forward-fix / backup-driven (no automatic downgrade).
  </action>
  <verify>Migrations execute successfully against external PostgreSQL without any DB container startup.</verify>
  <done>Schema upgrades are scripted and reproducible for production operations.</done>
</task>

<task type="auto">
  <name>Task 4: Implement DB bootstrap script (RBAC + departments + initial admin)</name>
  <files>scripts/prod/bootstrap_db.sh</files>
  <action>
Create `bootstrap_db.sh` that runs *after migrations* and is safe to re-run:
- seed RBAC contract:
  - `python scripts/seed_roles_permissions.py`
- seed departments:
  - `python scripts/seed_departments.py`
- bootstrap initial admin/CRO by email (SSO-safe pre-provision):
  - `python scripts/bootstrap_sso_user.py --email "$BOOTSTRAP_ADMIN_EMAIL" --role "$BOOTSTRAP_ADMIN_ROLE" --access-scope "$BOOTSTRAP_ADMIN_ACCESS_SCOPE"`

Run these commands in an ephemeral backend container using `--env-file backend.env` so DB access is always via external `DATABASE_URL`.
  </action>
  <verify>Bootstrap script completes successfully on a fresh DB and is idempotent on repeat runs.</verify>
  <done>RBAC and initial admin provisioning are automated, preventing SSO lockout.</done>
</task>

<task type="auto">
  <name>Task 5: Add idempotent bootstrap SSO user helper (backend/scripts)</name>
  <files>backend/scripts/bootstrap_sso_user.py</files>
  <action>
Implement a script that connects using `get_settings()` / `session_context(...)` and:
- validates role exists by name
- optionally resolves department (by code or name; make the interface explicit)
- creates the user if missing with:
  - `hashed_password=None`
  - `external_id=None` (so first SSO login can bind it)
  - `is_active=True`
- updates role/access_scope/department if user already exists (do not wipe `external_id` if set)
- exits non-zero with clear errors on invalid role/department/access scope

Do not print secrets. Avoid printing full connection strings.
  </action>
  <verify>`python backend/scripts/bootstrap_sso_user.py --help` runs and the script performs a safe idempotent upsert.</verify>
  <done>Initial admin bootstrap is possible without manual DB edits.</done>
</task>

</tasks>

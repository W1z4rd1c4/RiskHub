<objective>
Implement PI‑18‑03 reassessment cadence and reminders:
- Vendors supporting critical/important functions are reassessed at least **annually**.
- Other vendors are reassessed at least **every 3 years**.
- Major incidents or new threats can trigger an out-of-cycle reassessment.
</objective>

@backend/app/services/notification_service.py
</context>

<tasks>

<task type="auto">
  <name>Task 1: Data model — cadence + next due date</name>
  <action>
Extend vendor/assessment data to support scheduling:
- Store `reassessment_cadence_months` (12 or 36 by default based on critical/important flag).
- Store `next_reassessment_due_at` (deterministic; see rules below; editable only by privileged roles).
- Store `last_assessed_at` and `last_decided_at` on vendor for quick filtering.

Rule (documented):
- If `vendor.supports_important_core_insurance_function` then cadence=12 months else 36 months.
- Initial due date behavior (deterministic for dashboards/reports):
  - On vendor creation: set `next_reassessment_due_at = vendor.created_at + cadence` (even before the first assessment exists).
  - After an assessment reaches a final decision: set `next_reassessment_due_at = decision_at + cadence`.
  </action>
  <done>Each vendor has deterministic cadence and next due date fields that can be queried efficiently.</done>
</task>

<task type="auto">
  <name>Task 2: Scheduling service + notification triggers</name>
  <action>
Implement a service that:
- Computes due dates after an assessment reaches final decision.
- Finds vendors due soon/overdue (e.g., due within 30 days; overdue past due date).
- Emits notifications to outsourcing owner (and optionally Risk Manager) when due soon/overdue.

Implementation approach:
- Mirror the scheduler + dedupe patterns already used in questionnaire/KRI deadline services.
- Keep it safe if the scheduler runs multiple times (idempotent reminders; track `last_reassessment_reminded_at`).
- Use the vendor reassessment notification types defined in Plan 18-00:
  - `vendor_reassessment_due_soon`
  - `vendor_reassessment_overdue`

Localization + deep linking:
- Use backend i18n (`app.i18n.t`) for notification titles/messages (match `QuestionnaireDeadlineService` style; avoid hard-coded English strings).
- Store `resource_type="vendor"` and `resource_id=<vendor_id>` for all reassessment notifications.
  </action>
  <verify>Due soon/overdue reminders are not duplicated and are localized per user language.</verify>
  <done>Reassessment reminders reliably notify responsible users without spamming.</done>
</task>

<task type="auto">
  <name>Task 3: UI — schedule view + “start reassessment”</name>
  <action>
Add a “Schedule” tab on vendor detail:
- Show last assessment/decision date, next due date, cadence, status (ok/due soon/overdue).
- Provide CTA “Start reassessment” which creates a new draft assessment (reuses Plan 18-03).

UI scope decision:
- v1 includes the Vendor detail “Schedule” tab.
- A global “Upcoming Vendor Reviews” view is deferred to Phase 18 Dashboard/Committee surfaces (Plan 18-11).
  </action>
  <done>Users can see upcoming reviews and start reassessments from the UI.</done>
</task>

<task type="auto">
  <name>Task 4: Out-of-cycle triggers (minimal v1)</name>
  <action>
Support out-of-cycle reassessment triggers in a minimal, auditable way:
- Add `POST /vendors/{id}/trigger-reassessment` (privileged roles + outsourcing owner) that:
  - records a reason (incident/new threat/other)
  - sets `next_reassessment_due_at = now` and flags `reassessment_triggered_reason`

Full automation (incident → trigger) is handled in Plan 18-08.
  </action>
  <done>Major incidents/threats can force immediate reassessment with an audit trail.</done>
</task>

<objective>
Enable “comprehensive” TPRM by integrating external signals into vendor monitoring (PI‑18‑03 requires verifying public info and ongoing monitoring):
- Create a connector abstraction for external vendor risk signals.
- Implement at least one MVP connector (public registry/company data) and stubs for cyber rating and sanctions screening.
- Store signals with timestamps and optionally feed them into vendor risk scoring as a supplementary input (never replacing manual assessment).
</objective>

<tasks>

<task type="auto">
  <name>Task 1: Define a connector abstraction + storage model</name>
  <action>
Create:
- `VendorExternalSignal` model:
  - `vendor_id`
  - `provider_key` (public_registry, cyber_rating, sanctions)
  - `signal_type` (company_profile, risk_score, adverse_media, sanctions_hit, etc.)
  - `payload_json`
  - `fetched_at`
  - `status` (ok/error) + `error_message` (nullable)

Connector interface:
- `fetch(vendor) -> list[signals]`
- `supports(vendor) -> bool` (e.g., registry only if registration_id present)

Security:
- Keep provider secrets in env vars (no DB secrets unless encrypted storage already exists).
  </action>
  <done>Framework can store external signals from multiple providers in a consistent way.</done>
</task>

<task type="auto">
  <name>Task 2: Implement MVP provider + safe scheduling</name>
  <action>
Implement one real provider (MVP):
- `public_registry` connector that uses vendor’s registration_id and returns basic company facts (status, address, filings link).

Scheduling:
- Add a background refresh job that fetches signals periodically (e.g., daily/weekly), idempotent and rate-limited.
- Store last fetched time per vendor/provider to prevent duplication.
  </action>
  <verify>Provider failures are captured as signals with status=error, without breaking the rest of the system.</verify>
  <done>External signals can be fetched and refreshed reliably and safely.</done>
</task>

<task type="auto">
  <name>Task 3: API + UI — signals tab + manual refresh</name>
  <action>
Endpoints:
- `GET /vendors/{id}/signals`
- `POST /vendors/{id}/signals/refresh` (privileged roles + outsourcing owner)

Frontend:
- “Signals” tab showing:
  - latest signals grouped by provider
  - status and fetched time
  - manual refresh button gated by permissions

Optional (documented):
- feed selected signal outputs into vendor risk scoring as “supplementary indicators” (never auto-decide).
  </action>
  <done>Users can see and refresh external signals, and use them as evidence in assessments.</done>
</task>

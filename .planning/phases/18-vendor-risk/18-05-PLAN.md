<objective>
Implement PI‑18‑03 concentration and supply chain visibility:
- Model **vendor→vendor** relationships (subcontractors / fourth parties).
- Track which critical/important functions/services depend on which vendors.
- Compute and surface concentration risk signals (hard to replace; multiple critical functions with same/connected providers).
</objective>

<tasks>

<task type="auto">
  <name>Task 1: Data model — relationships + dependencies</name>
  <action>
Add:
- `VendorRelationship`:
  - `vendor_id` (primary vendor)
  - `related_vendor_id` (subcontractor / upstream dependency)
  - `relationship_type` (subcontractor, reseller, parent_company, other)
- `VendorService` (what the vendor provides):
  - `vendor_id`
  - `service_name` (free text)
  - optional notes/metadata as needed (keep v1 minimal)
- `VendorDependency` (what that service supports):
  - `vendor_service_id` (FK)
  - optional links to existing entities: `risk_id`, `department_id`
  - free-text `supported_function_name` (since there is no first-class Business Process entity)

Keep v1 pragmatic:
- If there is no “Business Process” entity, model dependencies via `risk_id` + `department_id` plus free-text `supported_function_name`.
- Relationship safety:
  - disallow self-relationships (`vendor_id == related_vendor_id`)
  - graph/aggregation code must be cycle-safe (track visited vendors, cap depth to avoid infinite loops)
  </action>
  <done>System can represent fourth parties and capture what critical/important functions depend on which vendor services.</done>
</task>

<task type="auto">
  <name>Task 2: Concentration risk analysis endpoint</name>
  <action>
Implement a service + endpoint that flags:
- Vendors marked “hard to replace”
- Vendors supporting important core insurance functions with:
  - multiple critical dependencies, or
  - dependencies in multiple departments, or
  - multiple ICT arrangements (service records) with same/connected providers

Return:
- a list of flags with human-readable reasons (keys + i18n handled in FE)
- a numeric “concentration risk score” (simple v1 heuristic is acceptable; document formula)
  </action>
  <done>Concentration risk is computed consistently and can be displayed and filtered.</done>
</task>

Do not block on perfect graph UX; prioritize correctness and data fidelity.
  </action>
  <done>Users can understand fourth-party exposure and critical dependency chains at a glance.</done>

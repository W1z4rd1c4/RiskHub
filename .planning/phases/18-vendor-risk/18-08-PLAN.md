<objective>
Implement PI‑18‑03 monitoring and escalation:
- Record ongoing monitoring outcomes (at least via incidents + remediation actions).
- Capture “major breaches” and “major security/operational incidents caused by ICT vendor” as structured records.
- Automatically trigger reassessment on major incidents (ties to Plan 18‑04).
- Track vendor SLA similarly to KRI (time-series values, breach states, reminders) without auto-mutating vendor score.
</objective>

<task type="auto">
  <name>Task 2: SLA tracking — KRI-like time series + breach logic</name>
  <action>
Implement vendor SLA as a first-class monitored entity **using the same mechanics as KRIs** (deep reuse of code patterns):

Backend model should be intentionally KRI-shaped to enable reuse of:
- historization (`*_value_history` table with period_start/period_end)
- breach status calculation (below/above/within)
- due-soon / overdue window logic
- scheduler + notification flow patterns

Data model (mirror KRI):
- `VendorSLA` (similar to `KeyRiskIndicator`):
  - `vendor_id` (FK)
  - `metric_name` (SLA name)
  - `description`
  - `current_value`
  - `lower_limit` and `upper_limit` (keep both even if SLA is one-sided; UI can guide which to use)
  - `unit`
  - `frequency` (daily/weekly/monthly/quarterly/annually)
  - `reporting_owner_id` (optional; fallback to `vendor.outsourcing_owner_user_id`)
  - `last_period_end`, `last_reported_at`
  - archive fields consistent with KRIs (`is_archived`, `archived_at`, `archived_by_id`)
  - computed property `breach_status` (below/above/within) identical semantics to KRI

- `VendorSLAValueHistory` (similar to `KRIValueHistory`):
  - `sla_id`
  - `period_start`, `period_end`
  - `recorded_at`
  - `recorded_by_id`
  - value snapshot (`value`, `lower_limit`, `upper_limit`, `unit`, `breach_status`)

Services (mirror KRI services by design):
- `VendorSLAHistoryService`: copy/adjust `KRIHistoryService` period logic and recording guards.
- `VendorSLADeadlineService`: copy/adjust `KRIDeadlineService` to create notifications for vendor SLAs.

Notifications:
- Add Vendor SLA notification types (parallel to KRI):
  - `vendor_sla_due_soon`, `vendor_sla_due_tomorrow`, `vendor_sla_overdue`
  - `vendor_sla_near_breach`, `vendor_sla_breach_detected`
- Add the enum/migration using the same pattern as prior notification enum extensions.
- Respect user notification preferences similarly to KRI where applicable.

Constraints:
- SLA breaches do NOT automatically change `vendor.risk_score_1_5` in MVP (manual score remains manual).
  </action>
  <verify>Creating an SLA value that breaches the threshold produces the expected breach state and notification(s) without duplication.</verify>
  <done>Vendors have KRI-like SLA monitoring with breach visibility and reminders.</done>
</task>

<task type="auto">
  <name>Task 3: API + UI — incidents, remediation, and SLA tabs</name>
  <action>
Endpoints:
- CRUD for incidents under vendor.
- CRUD for remediation actions (including assigning an owner).
- SLA endpoints mirroring KRI endpoints and shapes:
  - `GET /vendor-slas` (filters: vendor_id, breach_only, include_archived, paging)
  - `POST /vendor-slas`
  - `GET /vendor-slas/{id}`
  - `PUT /vendor-slas/{id}`
  - `DELETE /vendor-slas/{id}` (archive with reason)
  - `POST /vendor-slas/{id}/values` (record value)
  - `GET /vendor-slas/{id}/history`
  - `GET /vendor-slas/overdue` and `GET /vendor-slas/due-soon` (for dashboards)

Frontend:
- Add “Incidents” tab (log + filters + major badge).
- Add “Remediation” tab (action list with status/due dates and owner).
- Add “SLA” tab with:
  - SLA list + detail (definition) with the same form primitives as `KRIModal`
  - entry of values over time with the same period semantics as KRIs
  - breach badge + “due soon / overdue” visibility similar to existing KRI UX

RBAC:
- Outsourcing owner + privileged roles can create incidents.
- Owners can update remediation status; privileged roles can reassign.
For SLA:
- Read access: same as Vendor visibility (dept-scoped) + reporting owner cross-department.
- Write access: vendor `vendors:write` OR reporting owner (mirroring KRI patterns).
  </action>
  <done>Operational teams can record incidents and track remediation and SLA monitoring to completion.</done>
</task>

Ensure idempotency: multiple major incidents in short time do not spam or create inconsistent state.

  <done>Major vendor incidents escalate into reassessment and visibility automatically.</done>

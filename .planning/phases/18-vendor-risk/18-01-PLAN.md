<objective>
Implement the foundational **Vendor Catalog** required by PI‑18‑03:
- Create a single source of truth for third parties (vendors) with an assigned **outsourcing owner** (PI “vendor sponsor” / contract owner).
- Make Vendors follow the same *shape* as Risks (process/subprocess/department/owner), but with vendor-specific fields and a simpler score.
- Capture the **manual classification flags + vendor risk score** needed for TPRM governance:
  - important core insurance function support (yes/no)
  - DORA relevant (yes/no)
  - significant vendor (yes/no)
  - risk score **1–5** (manual) interpreted using the same OS‑18 impact → financial loss mapping used elsewhere in RiskHub.
- Provide CRUD API + UI with RBAC permissions consistent with RiskHub patterns.
</objective>

<clarifications>
Implementation conventions for Phase 18 (to avoid drift):
- **Permissions API:** use `require_permission()` / `check_permission()` from `backend/app/core/security.py` for permission checks, and `get_user_department_ids()` / `check_department_access()` from `backend/app/core/permissions.py` for department scoping (match `backend/app/api/v1/endpoints/risks.py` patterns).
- **Notifications:** vendor-related notifications must store `resource_type="vendor"` and `resource_id=<vendor_id>` (deep-linking handled in `NotificationBell` based on `notification.type`).
- **Activity log:** add VRM activity entity types in `ActivityEntityType` (consistent with existing `risk_questionnaire` / `kri_value`):
  - `vendor`, `vendor_assessment`, `vendor_incident`, `vendor_sla`, `vendor_remediation` (minimum set)
  - Vendor CRUD logs use `entity_type="vendor"` and `entity_id=<vendor_id>`
  - Sub-entity logs use `entity_id=<sub_entity_id>` with `department_id = vendor.department_id`
  - No DB migration is required for new activity entity types today because `activity_logs.entity_type` is stored as plain text.
- **Stable vendor reference:** v1 does **not** introduce a `vendor_id_code`. Reports/exports use vendor `id` + `name` as the identifier; name collisions are allowed.
- **Legal = compliance capability:** Phase 18 does not require a separate `legal` role; use `vendor_contracts:*` permission and grant it to `compliance` by default (Plan 18-00).
- **Admin safety:** unlike existing Risks/Controls list endpoints, Vendor read endpoints MUST enforce `vendors:read` to uphold the “admin has no business data” rule in `docs/BUSINESS_LOGIC.md`.
</clarifications>

<task type="auto">
  <name>Task 1: Backend data model — Vendor + criticality/materiality fields</name>
  <action>
Add a first-class `Vendor` entity (third party) that mirrors Risk’s core “shape” (process/subprocess/department/owner) while remaining vendor-specific:
- Identity: `name`, `legal_name?`, `registration_id?`, `country?`, `website?`
- Structure (match Risk field semantics where possible):
  - `process` (required; same meaning as Risk.process)
  - `subprocess` (optional; same meaning as Risk.subprocess)
  - `department_id` (nullable like Risks, but required by UI for non-privileged users; use the same access rules as Risks for null department)
- Ownership/governance:
  - `outsourcing_owner_user_id` (required; conceptually equivalent to Risk.owner_id; PI “Vlastník smlouvy je sponzor dodavatele”)
- Classification:
  - `vendor_type` (e.g., `ict`, `outsourcing`, `professional_services`, `partner`, `other`)
  - `risk_score_1_5` (int 1–5; fully manual)
  - `supports_important_core_insurance_function` (bool; PI “critical/important function” intent; fully manual)
  - `dora_relevant` (bool; fully manual)
  - `is_significant_vendor` (bool; fully manual; do not compute from materiality)
  - `materiality_assessed_max_impact_pct_own_funds` (optional numeric evidence input; PI references >4% materiality threshold)
  - `replaceability` (enum: easy/medium/hard) and `has_alternative_providers` (bool; for concentration risk later)
- Lifecycle: `status` (active/inactive), `created_at`, `updated_at`

Implementation notes:
- Prefer separate enums/tables only where needed; don’t over-normalize in v1.
- Add Alembic migration, ensure constraints:
  - outsourcing owner is required
  - department is nullable (like Risks), but `check_department_access(None, user)` should restrict unassigned vendors to privileged users only
- UI/UX interpretation of `risk_score_1_5` must reuse the existing OS‑18 impact scale helpers:
  - `IMPACT_DESCRIPTIONS[level]` for label/description
  - `formatFinancialRange(level, totalAssets)` (RiskHub `total_assets_value` config)
  </action>
  <done>DB schema supports outsourcing owner accountability + manual classification flags + risk score for each vendor.</done>

<task type="auto">
  <name>Task 3: Backend API — CRUD + filtering + department scoping</name>
  <action>
Create `backend/app/api/v1/endpoints/vendors.py` with endpoints:
- `GET /vendors` with Risk-like pagination patterns:
  - `skip`, `limit`, `search`, `sort_by`, `sort_order`
  - filters: `status`, `vendor_type`, `dora_relevant`, `supports_important_core_insurance_function`, `is_significant_vendor`, `outsourcing_owner_user_id`, `department_id`, `process`, `subprocess`, `risk_score_1_5`
- `POST /vendors`
- `GET /vendors/{id}`
- `PATCH /vendors/{id}`
- `DELETE /vendors/{id}` (or set inactive)

Scoping rules (match Risk patterns as closely as possible):
- Read gating MUST be explicit:
  - `GET /vendors` and `GET /vendors/{id}` require `vendors:read` (prevents Admin from accessing business data even if they have GLOBAL scope).
- Department scoping is enforced via `get_user_department_ids` / `check_department_access`:
  - AccessScope.GLOBAL users see all departments.
  - Non-privileged users are department-scoped by default.
- Cross-department ownership exception (match Risk owner behavior):
  - Outsourcing owner can read/edit their vendor regardless of department.
- List visibility MUST include cross-department owner visibility:
  - `GET /vendors` returns vendors in user departments OR where `outsourcing_owner_user_id == current_user.id` (to avoid “owner can’t find their vendor”).
- Existence leakage:
  - For `GET /vendors/{id}`, if the user is not authorized to read the vendor, return 404 (match Risk endpoints’ “don’t leak existence” pattern).
- Permission gating (match Risk patterns):
  - Create requires `vendors:write` + `check_department_access` (department heads get this permission via role).
  - Update requires `vendors:write` OR outsourcing owner (same as risks:write OR risk owner).
  - Delete/archive requires `vendors:delete` OR privileged policy choice; keep consistent with how risks/controls are handled.
  - Approvals: v1 does not route vendor edits/deletes through ApprovalRequest (explicitly deferred; see Phase 18 context).

Activity/Audit logging (match Risk endpoints):
- Log CREATE/UPDATE/ARCHIVE to Activity Log using `log_activity` and `build_change_set` under `ActivityEntityType.VENDOR`.

<objective>
Represent vendor risks aligned to PI‑18‑03:
- Implement a vendor risk taxonomy (regulatory, cyber, BCP, lock-in, etc.).
- Allow capturing vendor risk factors (category + notes) without over-engineering scoring.
- Enable **linking Vendors to enterprise Risks and Controls** so third‑party exposure is visible in the existing Risk Register and control landscape.
</objective>

<tasks>

<task type="auto">
  <name>Task 1: Define PI‑18‑03 vendor risk taxonomy (stable keys)</name>
  <action>
Create a stable taxonomy using keys (not localized strings) that match PI categories at minimum:
- `regulatory_legal`, `info_security_data`, `cyber_supply_chain`, `operational_continuity`,
  `service_quality`, `financial`, `strategic_reputational`, `governance_oversight`,
  `technology_lockin`, `human_factor`, plus `concentration` (DORA concentration checks).

Implementation choice:
- **Locked for v1:** use a constant list of stable `category_key` values (no DB table in v1).
- Localization lives in frontend i18n (`vendors.json`). Backend i18n is only needed for notifications (if any) and can reference the stable keys.
  </action>
  <done>Taxonomy keys exist and are ready to power forms, filters, and reports.</done>
</task>

<task type="auto">
  <name>Task 2: Backend model — VendorRiskFactors (no scoring) + link tables</name>
  <action>
Add simple `VendorRiskFactor` records (no per-factor scoring in MVP):
- `vendor_id`
- `category_key`
- `description`

Add link tables:
- `VendorRiskLink` (vendor ↔ enterprise risk, many-to-many)
- `VendorControlLink` (vendor ↔ control, many-to-many)

Constraints:
- Linking does not mutate vendor score automatically in MVP.
- Linking/unlinking requires authorization to both sides:
  - caller must be authorized to view/edit the Vendor, and to view/edit the linked Risk/Control (avoid leaking existence via cross-department links).
  </action>
  <done>Vendor risk factors are captured and Vendors can be linked to enterprise Risks and Controls.</done>
</task>

<task type="auto">
  <name>Task 3: Risk Register integration — vendor ↔ risk linking workflow</name>
  <action>
Keep Vendors separate from Risks:
- Do NOT automatically create enterprise `Risk` rows for each vendor.
- Implement explicit linking UI + API so a Vendor can be associated to multiple enterprise Risks.

Visibility:
- On Vendor detail, show linked Risks (with navigation).
- On Risk detail, show linked Vendors (with navigation).

RBAC/scoping (match existing cross-department ownership patterns):
- Vendor links must be visible to users who can see the Vendor (dept-scoped) and to the outsourcing owner cross-department.
- Risk detail should only show linked Vendors that the current user is authorized to see (avoid leaking vendor existence across departments).

Backend integration (match existing API style):
- Add `GET /risks/{risk_id}/vendors` to return linked vendors for a risk.
- Filter linked vendors by vendor visibility rules to avoid leaking cross-department vendor existence.

Frontend integration:
- Add a “Linked Vendors” section in `RiskDetailOverviewTab` showing the visible linked vendors with navigation.
  </action>
  <verify>A linked Vendor appears on Risk detail, and the linked Risk appears on Vendor detail.</verify>
  <done>Users can traverse Vendor ↔ Risk relationships without duplication.</done>
</task>

<task type="auto">
  <name>Task 4: API + UI — vendor factors + linked risks + linked controls</name>
  <action>
Add endpoints:
- Vendor risk factors:
  - `GET /vendors/{vendor_id}/risk-factors`
  - `POST /vendors/{vendor_id}/risk-factors`
  - `PATCH /vendor-risk-factors/{id}`
  - `DELETE /vendor-risk-factors/{id}` (or archive)

- Vendor ↔ Risk links:
  - `GET /vendors/{vendor_id}/linked-risks`
  - `POST /vendors/{vendor_id}/linked-risks` (link existing enterprise risk)
  - `DELETE /vendors/{vendor_id}/linked-risks/{risk_id}`

- Vendor ↔ Control links:
  - `GET /vendors/{vendor_id}/linked-controls`
  - `POST /vendors/{vendor_id}/linked-controls`
  - `DELETE /vendors/{vendor_id}/linked-controls/{control_id}`

Frontend:
- Add Vendor tabs:
  - “Risk Factors” (grouped by category)
  - “Linked Risks”
  - “Linked Controls”

UI reuse note (deep review against existing code):
- The app already has a dedicated linking UX for Risk↔Control: `LinkManagementDialog` + `LinkSearchPanel` + `ExistingLinksPanel`.
- For Vendor↔Risk and Vendor↔Control, prefer one of:
  1) Extend `LinkManagementDialog` to support vendor modes (recommended if it can be kept clean), or
  2) Create a `VendorLinkManagementDialog` that reuses `LinkSearchPanel`/`ExistingLinksPanel` patterns and API shapes.
Do not invent a completely new linking UX if we can reuse the existing one.
  </action>

# Plan 70-10: KRI Strict Submission Logic

## Goal
Implement a validation gate that enforces "Strict Period Submission" based on global configuration. This ensures data integrity for regulated environments where values must represent the exact end-of-period state.

## Technical Guidance
When a user submits a KRI value:
1.  **Strict Mode (Default)**: The system calculates the *current* reporting period based on the KRI's frequency (e.g., Monthly -> Jan 31, Feb 28). The user *cannot* choose a date; the system assigns the period end date automatically. Submission is rejected if the current date is too far past the period close (optional) or if they try to skip periods.
2.  **Open Mode**: The user can submit a value for *any* historical date. The system snaps it to the nearest period or accepts the raw date.

We will modify `record_kri_value` to check `kri_strict_period_submission`.

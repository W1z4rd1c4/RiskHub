# Plan 70-08: Global Configuration Foundation

## Goal
Establish the backend foundation for new configurable features by registering new properties in the Global Configuration system. This allows the application to maintain consistent defaults while allowing runtime overrides via the database.

## Technical Guidance
The RiskHub uses a hybrid configuration system:
1.  **Code Defaults (`ConfigDefaults`)**: Hardcoded fallback values used when the database is empty or the key is missing.
2.  **Database Overrides (`GlobalConfig` table)**: Runtime values set by the CRO/Admin.

**Why this layer?**
We need to introduce feature flags (`kri_strict_period_submission`, `control_execution_logging`) that modify business logic deep in the services. By centralizing them here, we ensure type safety (bool/int) and consistent default behavior across environments.

## Proposed Changes

### Backend

#### 1. Define Defaults
**File:** `backend/app/models/global_config.py`
**Action:** Add the following constants to the `ConfigDefaults` class.
```python
class ConfigDefaults:
    # ... existing defaults ...
    
    # KRI: Controls if submissions are rejected outside calculated period bounds
    KRI_STRICT_PERIOD_SUBMISSION = True
    
    # Controls: Feature flag to standardise or disable execution logging
    CONTROL_EXECUTION_LOGGING = True
```

#### 2. Seed Database Values
**File:** `backend/app/db/init_db.py`
**Action:** Update the `seed_global_config` function.
**Logic:**
- Iterate through the new keys.
- Use `get_global_config_value` to check existence.
- If missing, create a new `GlobalConfig` entry using the default from `ConfigDefaults`.
- **Do not overwrite** existing values (preserves user settings on restart).

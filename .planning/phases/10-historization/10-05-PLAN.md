# Plan 10-05: KRI Value Recording & Breach Detection

## Goal
Implement a robust backend endpoint for recording KRI values with automated breach detection and multi-recipient notifications.

## Proposed Changes

### [Backend] KRI Histortization Service

#### [MODIFY] [kris.py](../../../backend/app/api/v1/endpoints/kris.py)
- Create or update a `POST /kris/{id}/record` endpoint.
- Logic:
    1. Validate input value.
    2. Check reporting period constraints (lock if past 15 days of period end for non-privileged).
    3. Update `current_value` on the KRI model.
    4. Create a new `KRIValueHistory` entry.
    5. **Breach Detection**: Compare new value against `lower_limit` and `upper_limit`.
    6. **Notifications**: If breached, send immediate notifications to:
        - KRI Responsible Person
        - Linked Risk Owner
    7. Update `last_period_end` if it's a regular reporting submission.

### [Backend] Notifications System

#### [MODIFY] [notification_service.py](../../../backend/app/services/notification_service.py)
- Add a new notification type `KRI_BREACH_DETECTED`.
- Ensure multi-recipient logic is supported.

## Verification Plan

### Automated Tests
- Test cases for:
    - Normal value recording.
    - Breach triggering notifications to both owners.
    - Value within limits (no breach notification).
    - Submission window enforcement (15-day lock).
- Command: `cd backend && pytest tests/test_kri_historization.py`

### Manual Verification
1. Record a value outside limits through the UI (or Swagger).
2. Verify both KRI and Risk owners receive notifications.
3. Check the history log and current value update.

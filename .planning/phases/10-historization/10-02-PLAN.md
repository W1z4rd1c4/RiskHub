  <name>Task 1: Add KRI history service with reporting windows and reminder cadence</name>
  <files>backend/app/services/kri_history_service.py, backend/app/services/kri_deadline_service.py</files>
  <action>
    Create KRIHistoryService with helpers:
    - frequency_to_timedelta() mapping (daily=1, weekly=7, monthly=30, quarterly=90, annually=365)
    - current_period(kri): returns period_start/period_end based on last_period_end + frequency (rolling periods)
    - due_date(period_end): period_end + 15 days (reporting grace window); treat new period start as due_date + 1 day
    - reporting_owner(kri): kri.reporting_owner_id fallback to kri.risk.owner_id
    - record_value(db, kri, value, recorded_by_id=None, recorded_at=None, period_end=None) that:
      * enforces reporting window for non-privileged users (period_end must equal current period AND now <= due_date)
      * allows privileged users to backdate via explicit period_end
      * creates KRIValueHistory snapshot with period_start/period_end + value + limits + unit + breach_status
      * updates kri.current_value, kri.last_period_end, and kri.last_reported_at (recorded_at or now)
    
    Update KRIDeadlineService.check_kri_deadlines to:
    - compute period_end + due_date from last_period_end and frequency
    - send advance reminder before period_end (configurable constant, default 7 days)
    - send deadline notification on due_date (period_end + 15 days)
    - send overdue reminder every 7 weeks after due_date while not updated
    - route notifications to reporting_owner (fallback to risk.owner_id)
    - keep existing breach/near-breach logic intact, but keep message copy distinct (\"reporting due\" vs \"breach\")
    - de-duplicate overdue reminders using lookback_days=49
    - add counts in results dict: due_soon, deadline, overdue
  </action>
  <verify>cd backend && python -c "from app.services.kri_history_service import KRIHistoryService"</verify>
  <done>KRI history service enforces reporting windows and deadline checks include new reminder cadence</done>

  <files>backend/app/api/v1/endpoints/kris.py, backend/app/api/v1/endpoints/approvals.py</files>
  <action>
    Add endpoints:
    - POST /kris/{kri_id}/values: accepts KRIRecordValue (value, recorded_at optional),
      requires risks:write, validates department access, calls KRIHistoryService.record_value,
      blocks non-privileged backdating outside current period, returns updated KRIResponse.
    - GET /kris/{kri_id}/history: paginated list (page/size) with optional from_date/to_date filters,
      validates department access, returns KRIHistoryListResponse sorted by recorded_at desc.
    - GET /kris/overdue: list KRIs where due_date < now and last_period_end not updated,
      include due_date, days_overdue, reporting_owner for dashboard widget.
    - PATCH /kris/{kri_id}/history/{entry_id}: correction endpoint that:
      * Non-privileged users create ApprovalRequest (action_type EDIT, resource_type KRI)
        with pending_changes including history_entry_id + new value + reason.
      * Privileged users apply correction directly and update current_value if correcting latest entry.

    Update KRI create/update endpoints to:
    - accept reporting_owner_id and validate the user is in an accessible department
    - default reporting_owner_id to null (fallback handled by service layer)
    
    Update PUT /kris/{kri_id} to record history when current_value changes:
    - If update payload includes current_value, call record_value instead of direct assignment.
    - Avoid creating history entries when only metadata changes.
    
    Extend approval resolution for KRI edits:
    - If pending_changes includes history_entry_id, apply the correction to KRIValueHistory,
      and update current_value/last_period_end when the corrected entry is the latest period.

## Gating Findings

### 1) Medium — `/activity-log` is authenticated but not permission-gated; page still fetches and renders a misleading “empty” state on 403
- **Evidence:**
  - Route is accessible to any authenticated user (no permission wrapper): `frontend/src/App.tsx:81-116` (see `frontend/src/App.tsx:105`)
  - Page fetches options and logs on mount with no permission check:
    - `frontend/src/pages/ActivityLogPage.tsx:51-61`
    - `frontend/src/pages/ActivityLogPage.tsx:63-107`
  - On any error (including 403), UI falls back to “No activity logs found” (no access denied / error state): `frontend/src/pages/ActivityLogPage.tsx:221-225`
  - Backend correctly enforces `activity_log:read` (so no data leak, but UI still hits the endpoint): `backend/app/api/v1/endpoints/activity_log.py:19-33`
- **Impact:** Users without `activity_log:read` can still access the route directly and trigger repeated 403s (noise in logs, avoidable load) and receive a misleading “no data” UX that can be misread as “there are no logs” rather than “you don’t have access”.
- **Suggested fix:** Add a permission-gated route wrapper (or gate inside `ActivityLogPage`) using `canViewActivityLog`, and render an explicit “Access denied” view without calling `activityLogApi` when permission is missing. Consider differentiating empty results vs error/403 states.

### 2) Medium — Admin navigation hides Activity Log even though `admin` is seeded with full permissions (inconsistent access model)
- **Evidence:**
  - Admin-only navigation excludes all business pages (including Activity Log): `frontend/src/components/layout/Sidebar.tsx:124-145` (admin branch at `frontend/src/components/layout/Sidebar.tsx:128-135`)
  - Activity Log link is only added in the “business user” navigation branch: `frontend/src/components/layout/Sidebar.tsx:136-145` (see `frontend/src/components/layout/Sidebar.tsx:141`)
  - Permission check itself is role-agnostic (`hasPermission('activity_log','read')`): `frontend/src/hooks/usePermissions.ts:12-46` (see `frontend/src/hooks/usePermissions.ts:42-43`)
  - Role seed grants `admin` full access (`*:*`): `backend/app/db/seed.py:47-56` (see `backend/app/db/seed.py:48-55`)
- **Impact:** Admins can access `/activity-log` by URL (and the backend will allow it), but they cannot discover it via navigation. This is confusing during compliance reviews and indicates an unresolved product decision: “admin is separate console only” vs “admin is privileged business visibility”.
- **Suggested fix:** Decide whether `admin` should (A) see business data like Activity Log (then include the link in admin nav), or (B) be restricted from business data (then remove `*:*` or explicitly exclude `activity_log:read` and/or block the route for `admin`).

## UI & API Alignment Findings

### 3) High — Diff rendering assumes `{ old, new }` per field; can crash on null/mismatched shapes and also misrepresents falsy values
- **Evidence:**
  - UI assumes each `entry.changes[field]` has `old` and `new`, and uses `delta.old || 'none'` / `delta.new || 'none'`: `frontend/src/pages/ActivityLogPage.tsx:259-272`
  - Backend currently logs KRI updates with `changes=update_data` (new-values-only, values may be primitives or `None`): `backend/app/api/v1/endpoints/kris.py:377-409` (see `backend/app/api/v1/endpoints/kris.py:399-408`)
  - Backend response schema does not constrain `changes` to the `{old,new}` shape: `backend/app/schemas/activity_log.py:7-20` (see `backend/app/schemas/activity_log.py:17`)
- **Impact:**
  - If any `changes` value is `null` (or another unexpected shape), the Activity Log page can throw at render time (`delta.old`) and become unusable for auditors.
  - Even when it doesn’t crash, `|| 'none'` causes incorrect display for legitimate falsy values (e.g., `0`, `false`, `''`).
- **Suggested fix:** Make the frontend renderer defensive (handle primitives/nulls, prefer `??` over `||`, and render JSON safely for objects). Align the backend to a single `changes` contract (recommended: `{field: {old, new}}`) and/or version the contract.

### 4) Medium — Date range filters use date-only strings while backend expects `datetime`; `date_to` likely excludes same-day events
- **Evidence:**
  - UI uses `<input type="date">` for `dateFrom/dateTo`: `frontend/src/pages/ActivityLogPage.tsx:194-212`
  - Filters send `date_from` / `date_to` directly as strings: `frontend/src/pages/ActivityLogPage.tsx:86-94`
  - Backend expects `datetime` and applies `created_at <= date_to`: `backend/app/api/v1/endpoints/activity_log.py:32-33` and `backend/app/api/v1/endpoints/activity_log.py:73-76`
- **Impact:** Selecting an end date (e.g., `2026-01-04`) will be interpreted as midnight at the start of that date, excluding all events later that same day, which can cause audit gaps and user confusion.
- **Suggested fix:** Treat `date_to` as end-of-day in the frontend (append `T23:59:59.999Z` / local equivalent) or adjust backend semantics to interpret date-only `date_to` as inclusive end-of-day (or implement “next day exclusive” filtering).

### 5) Medium — Pagination renders O(number_of_pages) elements, which will degrade badly as Activity Logs grow
- **Evidence:** `frontend/src/pages/ActivityLogPage.tsx:300-315` creates `Array.from({ length: Math.ceil(total / limit) })` and iterates over every page index.
- **Impact:** Activity Logs are expected to grow quickly; rendering/iterating thousands of pages on every render can noticeably slow or freeze the UI even though only a small subset of page buttons is actually displayed.
- **Suggested fix:** Generate a small “window” of page buttons (first/last/current±1) without building an array the size of `total/limit`, or use a pagination component that renders a bounded set.

### 6) Low — Entry label rendering has no fallback for unknown action/entity values
- **Evidence:** `frontend/src/pages/ActivityLogPage.tsx:245-248` renders `ACTION_LABELS[entry.action]` and `ENTITY_TYPE_LABELS[entry.entity_type]` without fallbacks.
- **Impact:** If the backend introduces a new enum value, the UI can show blank labels (hurting audit readability).
- **Suggested fix:** Add `|| entry.action` / `|| entry.entity_type` fallbacks (matching the dropdown behavior at `frontend/src/pages/ActivityLogPage.tsx:187-190`).

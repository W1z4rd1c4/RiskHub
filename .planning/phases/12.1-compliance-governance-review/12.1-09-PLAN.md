<task type="checkpoint:decision" gate="blocking">
  <decision>Define truthful semantics for quarter-over-quarter metrics (period vs snapshot)</decision>
  <context>
Audit finding: many “quarterly comparison” metrics ignore the (start,end) window and are therefore identical across quarters (misleading).

We need to choose a consistent approach:
- Period metrics (count events within the window)
- Snapshot metrics (state as-of a point-in-time) — requires history/snapshots to be accurate
- Mixed but explicitly labeled (period-only in the comparison widget; snapshot-only elsewhere)
  </context>
  <options>
    <option id="period-only">
      <name>Quarterly comparison shows ONLY metrics that are truly window-based</name>
      <pros>Truthful without needing historical tables; easier to test.</pros>
      <cons>May reduce the number of metrics shown in the widget.</cons>
    </option>
    <option id="mixed-explicit">
      <name>Split into two groups: period comparison + current snapshot (clearly labeled)</name>
      <pros>Retains useful “current posture” metrics while keeping comparisons truthful.</pros>
      <cons>Requires API + UI refactor to render separate sections and avoid misleading deltas.</cons>
    </option>
    <option id="historical-snapshots">
      <name>Implement true quarter-end snapshots (persisted) to compare snapshots accurately</name>
      <pros>Best executive accuracy.</pros>
      <cons>Largest scope: new table/job, backfill strategy, and more complexity.</cons>
    </option>
  </options>
  <resume-signal>Select: `period-only`, `mixed-explicit`, or `historical-snapshots`.</resume-signal>
</task>

Implement the selected semantics:

If `period-only`:
- Only include metrics that are verifiably within `[start,end)` using timestamped tables.
- Remove snapshot metrics from `changes` computation (or remove them entirely from the quarterly endpoint response).

If `mixed-explicit`:
- API: return `{ period_metrics: {this,last,changes}, snapshot_metrics: {...} }` or similar, and update the widget accordingly.
- UI: render distinct sections so snapshot metrics are not presented as quarter deltas.

If `historical-snapshots`:
- Introduce a snapshot storage mechanism (table + capture job) and update endpoint to read snapshots for each quarter.

Also fix “Closed risks” semantics:
- Current logic uses `updated_at` + `status == closed` which does NOT mean “closed in period”.
- Recommended: derive “closed in period” from a status-change event (Activity Log `changes.status`) once Activity Log diffs are fixed (Plan 12.1-07).

Also fix enum drift risks:
- Replace string-literal approval status filtering (e.g., `["pending","pending_privileged"]`) with `ApprovalStatus` enum values (or enum-typed comparisons) so metrics don’t silently break when statuses evolve.

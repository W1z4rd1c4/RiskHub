  <files>backend/app/api/v1/endpoints/risks.py, backend/app/api/v1/endpoints/controls.py, backend/app/api/v1/endpoints/kris.py, backend/app/core/activity_logger.py</files>
  <action>
Fix incorrect diff capture and enforce a single contract for `changes`:
`{ field_name: { old: <value>, new: <value> } }`.

Implementation guide:
- Create a small helper in `backend/app/core/activity_logger.py` (or a new module in `backend/app/core/`) to build `changes` safely:
  - Accept `(model_instance, update_data_dict)` and return a `{field: {old,new}}` dict.
  - Snapshot old values BEFORE mutation (do not read “old” values after commit/refresh).
  - Normalize enums to their `.value`.
  - Exclude fields that should not be logged (e.g., passwords, hashed_password).
  - Use `None` vs missing consistently (omit keys not changed).
- Update:
  - `backend/app/api/v1/endpoints/risks.py` update path: capture `old_values` before applying updates; build `changes` after applying updates using old snapshot + new values.
  - `backend/app/api/v1/endpoints/controls.py` update path similarly.
  - `backend/app/api/v1/endpoints/kris.py`: replace `changes=update_data` (new-only) with `{old,new}` diffs.

Important pitfalls:
- Recalculated scores (gross/net) can change implicitly; decide whether to log them explicitly (recommended: include if user changed probability/impact).
- Avoid `delta.old || 'none'` style falsy bugs by keeping raw values in the API; frontend can format.
  </action>

<task type="checkpoint:decision" gate="blocking">
  <decision>Choose an Activity Log search strategy that scales (SQLite tests vs Postgres production)</decision>
  <context>
Audit findings require two things at once:
- Search should support “values” (i.e., include `changes`) as described in Phase 12 context.
- Search should not degrade as the Activity Log table grows (current multi-column `%term%` ILIKE is a classic full-scan risk).

Repo constraint: backend tests use SQLite in-memory (`backend/tests/conftest.py`), while production is likely Postgres. The implementation must therefore either (A) stay cross-DB and mitigate scans via bounded defaults, or (B) use a Postgres-optimized path with a safe SQLite fallback.
  </context>
  <options>
    <option id="default-window-ilike">
      <name>Cross-DB: add `changes` to search + apply a default time window when `search` is used</name>
      <pros>
- Works on SQLite tests and any DB backend.
- Minimal migration risk.
      </pros>
      <cons>
- Not “true” full-text; substring matching is still expensive for very large windows.
- Requires product acceptance that “search defaults to recent activity” unless a date range is provided.
      </cons>
    </option>
    <option id="postgres-fulltext">
      <name>Postgres: full-text search (tsvector + GIN) with SQLite fallback</name>
      <pros>
- Production-grade performance.
- Closest match to “fulltext across change data” intent.
      </pros>
      <cons>
- Requires migrations and some DB-specific SQL.
- CI/tests won’t fully validate the Postgres path unless you add a Postgres test job.
      </cons>
    </option>
    <option id="postgres-trigram">
      <name>Postgres: trigram indexes (pg_trgm) for substring search with SQLite fallback</name>
      <pros>
- Keeps substring semantics (user-friendly).
- Often simpler than full-text ranking for “find a value anywhere”.
      </pros>
      <cons>
- Requires enabling extension + DB-specific indexes.
- Still needs guardrails for pathological queries.
      </cons>
    </option>
  </options>
  <resume-signal>Select: `default-window-ilike`, `postgres-fulltext`, or `postgres-trigram`.</resume-signal>
</task>

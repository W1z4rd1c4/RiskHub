<task type="auto">
  <name>Task 3: Make diff rendering robust to real-world `changes` shapes and falsy values</name>
  <files>frontend/src/pages/ActivityLogPage.tsx, frontend/src/types/activityLog.ts</files>
  <action>
Fix the brittle diff renderer which assumes `{old,new}` for every field and incorrectly maps falsy values to “none”.

Implementation guide:
- Update the renderer to handle:
  - `changes === null` (no diff section)
  - `{field: {old,new}}` (happy path)
  - `{field: <primitive>}` or `{field: null}` (legacy/misaligned backend) without crashing
- Use `??` instead of `||` when applying fallbacks so `0/false/""` are preserved.
- Add fallbacks for unknown `action`/`entity_type` label mapping:
  - Render `ACTION_LABELS[entry.action] ?? entry.action`
  - Render `ENTITY_TYPE_LABELS[entry.entity_type] ?? entry.entity_type`
- Add truncation/pretty-printing for objects/arrays:
  - render JSON in a `<pre>`-style block for large values (or a tooltip/popover)
  - cap maximum rendered length per value to avoid UI blowups on large payloads

Keep rendering safe (no `dangerouslySetInnerHTML`).
  </action>
  <verify>cd frontend && npm run test:run</verify>
  <done>Activity Log UI never crashes on unexpected `changes` shapes and displays falsy values correctly.</done>

  <files>frontend/src/pages/ActivityLogPage.tsx, frontend/src/services/activityLogApi.ts</files>
  <action>
Fix the “end date excludes same-day activity” bug.

Implementation guide:
- The UI uses `<input type="date">` which yields `YYYY-MM-DD`.
- Backend expects `datetime` and currently applies `created_at <= date_to`.
- Convert `date_to` into an inclusive end-of-day timestamp before sending:
  - For local time: append `T23:59:59.999` (and be consistent about timezone), OR
  - Safer: send an exclusive upper bound by adding 1 day and using `< next_day` (requires backend support).

Choose one approach and ensure the UI and backend align.

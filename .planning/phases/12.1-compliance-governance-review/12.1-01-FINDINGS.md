## Scope Alignment

### 1) High — Risk Committee endpoints lack backend privileged gating (UI-only protection)
- **Evidence:**
  - `backend/app/api/v1/endpoints/dashboard.py:607-611` and `backend/app/api/v1/endpoints/dashboard.py:835-839` use `current_user: User = Depends(deps.get_current_user)` with no privileged/permission gate.
  - UI hides the Risk Committee view for non-privileged users (`frontend/src/pages/DashboardPage.tsx:200-227`), but that does not protect the API.
- **Impact:** Any authenticated user can call committee endpoints directly and receive committee-only metrics (critical risks, department exposure, recent activity), which violates the “privileged only” expectation implied by the UI and Phase 12 intent.
- **Suggested fix:** Add backend authorization (e.g., require global scope / privileged role / dedicated permission) to both endpoints, and add regression tests to prevent reintroduction.

### 2) Medium — Activity Log UI does not match Phase 12 “view modes” and search scope
- **Evidence:**
  - Phase 12 required view modes include By Person / By Department / By Risk / By Entity Type (`.planning/phases/12-compliance-governance/12-CONTEXT.md:27-35`).
  - Current UI uses entity-type tabs and limited filters only (`frontend/src/pages/ActivityLogPage.tsx:27-43`, `frontend/src/pages/ActivityLogPage.tsx:63-103`). There is no actor/department/risk-specific view mode or direct “by risk” drilldown.
  - Backend search is limited to `description`, `entity_name`, and `actor_name` (`backend/app/api/v1/endpoints/activity_log.py:64-72`), not “values” from `changes` as described in the vision (`.planning/phases/12-compliance-governance/12-CONTEXT.md:35`).
- **Impact:** Review/audit workflows described for Phase 12 (e.g., “show me everything by Person X” or “within Department Y”) are not directly supported by the UI and only partially supported by backend filters.
- **Suggested fix:** Either (A) implement the defined view modes + actor/department/risk filtering UX and expand search to include `changes` (carefully, for performance), or (B) update Phase 12 vision docs to reflect the delivered UX.

### 3) Medium — Activity Log visibility for Admin role is inconsistent with Phase 12 access-control note
- **Evidence:**
  - Phase 12 states Activity Log should be visible to Admin (`.planning/phases/12-compliance-governance/12-CONTEXT.md:53-55`).
  - Admin navigation intentionally excludes business navigation (including Activity Log), rendering only Settings/Access/Admin/Docs (`frontend/src/components/layout/Sidebar.tsx:124-145`).
  - Admin default landing redirects to `/admin` (`frontend/src/App.tsx:61-67`).
- **Impact:** Admins may not discover Activity Log in the UI despite having the intended permission; this can read as “not shipped” or “not accessible” during reviews.
- **Suggested fix:** Decide whether Admin is intended to be a privileged business user for Activity Log. If yes, add Activity Log to admin navigation; if no, update Phase 12 access-control note to reflect the product’s separation between Admin Console and business views.

### 4) Medium — Risk Committee “Quarterly Report Package” is defined in Phase 12 vision but not implemented/tracked
- **Evidence:** The vision lists “Quarterly Report Package” (`.planning/phases/12-compliance-governance/12-CONTEXT.md:13-16`), but no other code/planning artifacts reference it (search only finds the context line).
- **Impact:** One of the declared Risk Committee deliverables is missing or undocumented; reviewers cannot confirm completion.
- **Suggested fix:** Add an explicit plan/implementation for “Quarterly Report Package” (define what it is: PDF pack, export bundle, slide deck, etc.), or remove it from Phase 12 vision if it is not required.

## Out-of-Scope Drift

### 5) Low — Per-entry “old → new” diffs resemble the out-of-scope “side-by-side diff/comparison view”
- **Evidence:**
  - Out-of-scope explicitly includes “Side-by-side diff/comparison view” (`.planning/phases/12-compliance-governance/12-CONTEXT.md:39-45`).
  - Activity Log UI renders field-level old/new values side-by-side (`frontend/src/pages/ActivityLogPage.tsx:259-276`).
  - Phase 12-02 summary also calls out “side-by-side diffs” (`.planning/phases/12-compliance-governance/12-02-SUMMARY.md:17-21`).
- **Impact:** If “comparison view” was meant to prohibit any side-by-side diff visualization, this violates scope. If it was meant to prohibit time-range comparison tooling (not per-entry diffs), this is acceptable but should be clarified.
- **Suggested fix:** Clarify “comparison view” definition in Phase 12 context. If disallowed, remove per-entry diffs and keep descriptions only.

## Planning Metadata & Evidence

### 6) High — Admin log endpoints read the wrong log file path vs logging configuration
- **Evidence:**
  - Logging configuration writes to `backend/logs/*.json.log` (`backend/app/core/logging.py:96-102`).
  - Admin endpoints attempt to read from `backend/app/logs/*.json.log` (`backend/app/api/v1/endpoints/admin.py:486-488`, `backend/app/api/v1/endpoints/admin.py:571-573`).
  - Phase 12-05/12-06 summaries claim admin log endpoints are functional (`.planning/phases/12-compliance-governance/12-05-SUMMARY.md:39-50`, `.planning/phases/12-compliance-governance/12-06-SUMMARY.md:44-50`), but the file path mismatch strongly suggests the UI endpoints will return empty results unless logs are manually placed in the unexpected directory.
- **Impact:** Admin Console “Application Logs” and “Audit Logs” tabs may show no data, undermining SIEM verification and admin operational workflows.
- **Suggested fix:** Centralize log file path resolution (single helper in logging module) and reuse it in admin endpoints; add a small regression test that asserts the endpoint reads from the configured log directory.

### 7) Medium — ROADMAP vs STATE inconsistencies for Phase 12 tracking
- **Evidence:**
  - ROADMAP marks Phase 12 as `Status: Planning` and `Plans: 3 plans` while listing 6 completed plans (`.planning/ROADMAP.md:236-249`).
  - STATE marks Phase 12 as complete (6/6) but Phase 12 progress list omits 12-07 (`.planning/STATE.md:73-86`).
  - Phase directory contains `12-04-*` artifacts but ROADMAP Phase 12 plan list does not mention 12-04 (`.planning/ROADMAP.md:243-249`).
- **Impact:** Reviewers cannot rely on ROADMAP/STATE for accurate completion status; this increases audit cost and risk of missing work.
- **Suggested fix:** Update ROADMAP/STATE to reflect the real plan set (including 12-04 and 12-07), correct plan counts, and ensure Phase 12 status is consistent with completion claims.

### 8) Medium — Phase 12-07 documentation references and claims drift from implementation
- **Evidence:**
  - 12-07 summary references `backend/docs/SIEM_INTEGRATION.md` (`.planning/phases/12-compliance-governance/12-07-SUMMARY.md:26`), but documentation is currently in `backend/docs/ADMIN_SIEM_GUIDE.md` (file exists; referenced file does not).
  - 12-07 summary claims “Built-in search and PDF export features for documentation” (`.planning/phases/12-compliance-governance/12-07-SUMMARY.md:16-20`), but the documentation UI is a simple library/detail view with no search/export (`frontend/src/pages/DocumentationPage.tsx:70-109`).
- **Impact:** Summary is not a reliable evidence source; reviewers may chase non-existent files/features.
- **Suggested fix:** Align docs filenames/paths with summaries and admin endpoint expectations; update the summary to reflect actual features (or implement search/export if it is truly required).

### 9) Medium — Missing regression tests for Phase 12 security and scoping expectations
- **Evidence:**
  - Testing expectations are defined (`.planning/codebase/TESTING.md`), and Phase 12-01 even calls out a missing `backend/tests/test_activity_log.py` as a next step (`.planning/phases/12-compliance-governance/12-01-SUMMARY.md:181-186`).
  - No backend tests found covering: Activity Log RBAC + department scoping, Risk Committee endpoint authorization, or Admin logs path correctness.
- **Impact:** Security regressions (like endpoint auth gaps) and operational regressions (like log path mismatches) are likely to recur unnoticed.
- **Suggested fix:** Add targeted pytest tests for the three critical surfaces: `/activity-log` scoping, `/dashboard/*committee*` authorization, and `/admin/logs/*` correctness.

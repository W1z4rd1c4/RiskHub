## Backend Logging/Admin Findings

### 1) High — Admin file-log endpoints read the wrong log directory vs logging configuration
- **Evidence:**
  - Logging writes to `backend/logs/*.json.log`: `backend/app/core/logging.py:96-102`
  - Admin endpoints read from `backend/app/logs/*.json.log`:
    - `backend/app/api/v1/endpoints/admin.py:486-488` (`/admin/logs/recent`)
    - `backend/app/api/v1/endpoints/admin.py:571-573` (`/admin/logs/audit`)
- **Impact:** Admin Console “Recent Logs” / “Audit Logs” will show empty results (or misleading file paths) unless logs are manually duplicated into an unexpected directory. This also undermines the SIEM guide’s stated log location.
- **Suggested fix:** Centralize log path resolution in `backend/app/core/logging.py` (single helper returning `backend/logs`) and reuse it in admin endpoints. Add a small regression test asserting admin endpoints read the configured directory.

### 2) High — Admin “technical logs” and “session revoke” auditing appear broken (field/signature mismatch)
- **Evidence:**
  - ActivityLog has `description` and no `details`: `backend/app/models/activity_log.py:66-71`
  - `/admin/logs` response builds `details=log.details`: `backend/app/api/v1/endpoints/admin.py:322-333`
  - “Revoke session” logs with non-existent `user_id`/`details` args for `log_activity`: `backend/app/api/v1/endpoints/admin.py:423-435`
  - `log_activity` signature expects `actor` and `description`, not `user_id`/`details`: `backend/app/core/activity_logger.py:11-22`
- **Impact:** The affected admin endpoints likely 500 at runtime (AttributeError / TypeError), making SIEM/admin operational tooling unreliable.
- **Suggested fix:** Align admin endpoints to the real ActivityLog contract:
  - Use `description` instead of `details`
  - Call `log_activity(..., actor=current_user, description=...)`
  - Ensure the endpoint commits after logging where required.

### 3) Medium — JWT `user_id` context is not populated (wrong claim parsed) and can be spoofed for rejected requests
- **Evidence:**
  - Middleware extracts `user_id` from JWT `sub` and casts to int: `backend/app/middleware/logging_context.py:91-112`
  - Tokens encode `sub` as email and `user_id` as numeric id: `backend/app/api/v1/endpoints/auth.py:77-78`
  - Middleware explicitly disables signature verification for speed: `backend/app/middleware/logging_context.py:105-110`
- **Impact:** `user_id` is likely missing from many logs (or misattributed on invalid tokens), weakening SIEM correlation and forensic value.
- **Suggested fix:** Extract `user_id` from the `user_id` claim (not `sub`). Consider verifying signature (or only setting `user_id` after successful auth) to prevent misattribution on rejected requests.

### 4) Medium — Rotation config is persisted, but backend logging reads only a cache and likely ignores DB config on restart
- **Evidence:**
  - Logging reads rotation settings from `get_config_sync(...)` (cache-only): `backend/app/core/logging.py:60-76`
  - `get_config_sync` never queries DB (cache only): `backend/app/models/global_config.py:177-190`
  - Admin endpoint claims restart required and persists values: `backend/app/api/v1/endpoints/admin.py:715-766`
  - Migration seeds defaults (10MB × 10): `backend/alembic/versions/72d3046593d5_add_log_rotation_config.py:21-29`
- **Impact:** Operators may change log rotation via `/admin/logs/config` and restart, but the file handlers still use defaults because config is never loaded into cache before logging is configured. This creates “false control” risk in compliance/ops.
- **Suggested fix:** Either (A) move rotation config to environment variables (and document it), or (B) load global config on startup before configuring logging, or (C) redesign `get_log_settings()` to read from DB safely during startup.

### 5) Medium — Log file endpoints are potentially expensive (they scan the entire log file on every request)
- **Evidence:**
  - `/admin/logs/recent` reads the full file line-by-line and only keeps a deque tail: `backend/app/api/v1/endpoints/admin.py:500-504`
  - `/admin/logs/audit` does the same: `backend/app/api/v1/endpoints/admin.py:585-589`
- **Impact:** As logs grow, these endpoints become slower and increase I/O load. This can degrade Admin Console responsiveness during incidents (when log access is most needed).
- **Suggested fix:** Implement true tail reading (seek from end and read chunks until N lines), or maintain a rolling index/offset. Consider server-side pagination rather than “last N lines”.

## Docs & Frontend Integration Findings

### 6) Medium — SIEM guide schema does not match actual emitted fields (e.g., `actor_id` vs `user_id`)
- **Evidence:**
  - Guide claims `user_id` is present and `logger` is always `audit`: `backend/docs/ADMIN_SIEM_GUIDE.md:21-36`
  - Actual audit logger emits `actor_id` / `actor_name` and entity fields: `backend/app/core/activity_logger.py:56-69`
  - Request context may not populate `user_id` due to middleware claim mismatch: `backend/app/middleware/logging_context.py:91-112`
- **Impact:** SIEM ingestion mappings and dashboards built from the guide will be wrong (miss user attribution, mislabel fields), increasing operational friction during audits/incidents.
- **Suggested fix:** Update the guide with an example real audit log line and a definitive field list (including `actor_id`, `entity_type`, `entity_id`, etc.). Clarify when `request_id`/`client_ip` are present.

### 7) Medium — Failed login auditing is misclassified and leaks user existence details into logs
- **Evidence:**
  - Failed login attempts use `ActivityAction.LOGIN` (not `FAILED_LOGIN`) and include distinct failure descriptions:
    - Not found/no password: `backend/app/api/v1/endpoints/auth.py:43-56`
    - Wrong password: `backend/app/api/v1/endpoints/auth.py:58-71`
  - `ActivityAction.FAILED_LOGIN` exists: `backend/app/models/activity_log.py:21-23`
  - Admin technical logs try to treat `failed_login` as higher severity, but will not match if everything is `login`: `backend/app/api/v1/endpoints/admin.py:323-331`
- **Impact:** SIEM rules and admin views cannot reliably separate successful vs failed logins; logs also explicitly reveal “user exists” vs “not found” (useful for forensic detail, but higher exposure if logs are ever accessed improperly).
- **Suggested fix:** Emit `failed_login` for failures (and keep responses to clients generic). Consider keeping detailed failure reason in structured fields with appropriate access controls.

### 8) Low — `verify_audit_logs.py` does not verify key invariants it documents (and defines unused `AUDIT_FIELDS`)
- **Evidence:**
  - Defines `AUDIT_FIELDS` but does not enforce them: `backend/scripts/verify_audit_logs.py:12-16`
  - Only checks `REQUIRED_FIELDS` and secret regex warnings: `backend/scripts/verify_audit_logs.py:46-57`
- **Impact:** The script can pass even when audit logs are missing expected audit context (e.g., `client_ip`) or when non-audit entries leak into the audit stream. This reduces confidence in the verification step.
- **Suggested fix:** Expand the script to (A) require `logger == "audit"` for audit logs, (B) validate presence of key audit fields (and handle legitimate nulls), and (C) optionally validate separation by checking that `app.json.log` contains no `logger == "audit"` entries.

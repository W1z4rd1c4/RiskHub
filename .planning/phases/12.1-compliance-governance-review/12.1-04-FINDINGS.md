## Scope
- **Backend committee endpoints:** `backend/app/api/v1/endpoints/dashboard.py`, `backend/app/api/deps.py`, `backend/app/core/permissions.py`

## Summary
- **Critical:** 0

## Backend Findings

### 1) High — Risk Committee endpoints are not privileged-only on the backend (UI-only protection)
- **Evidence:**
  - Endpoints use `Depends(deps.get_current_user)` with no privileged/permission gate:
    - `backend/app/api/v1/endpoints/dashboard.py:607-611` (`/dashboard/quarterly-comparison`)
    - `backend/app/api/v1/endpoints/dashboard.py:835-839` (`/dashboard/committee-summary`)
  - `deps.get_current_user` authenticates but does not enforce privileged visibility: `backend/app/api/deps.py:18-89`
  - Privileged concept exists (`is_privileged_user` → global scope), but is not applied here: `backend/app/core/permissions.py:42-45`
- **Impact:** Any authenticated user can call committee endpoints directly and receive privileged-only data (including top risks, department exposure, and system-wide metrics), even if the UI hides the Risk Committee view.
- **Suggested fix:** Add backend authorization to both endpoints (e.g., require global scope via `is_privileged_user(current_user)` and/or introduce a dedicated permission like `dashboard:committee` enforced with `require_permission(...)`). Add regression tests to prevent reintroduction.

### 2) Medium — Quarter boundary logic likely undercounts the last day of the previous quarter (and uses naive datetimes)
- **Evidence:**
  - `last_quarter_end = current_quarter_start - timedelta(days=1)` produces midnight at the start of the last day of the quarter: `backend/app/api/v1/endpoints/dashboard.py:625-631`
  - The endpoint uses `datetime.now()` without timezone awareness: `backend/app/api/v1/endpoints/dashboard.py:625`
- **Impact:** “Last quarter” window can exclude most of the last day’s activity, producing systematic undercounts and misleading quarter-over-quarter changes. Naive timestamps can also cause timezone-related confusion if DB timestamps are stored with timezone.
- **Suggested fix:** Use an end-exclusive window (`created_at >= start AND created_at < next_period_start`) or compute an inclusive end-of-day timestamp precisely (microsecond precision). Prefer timezone-aware UTC datetimes consistently.

### 3) Medium — Several “quarterly comparison” metrics ignore the `(start, end)` window, making `this_quarter` vs `last_quarter` misleading
- **Evidence:** The following metrics do not use `start/end` at all, so they will be identical in `this_quarter` and `last_quarter`:
  - `priority_risks`: `backend/app/api/v1/endpoints/dashboard.py:663-669`
  - `kri_breaches`: `backend/app/api/v1/endpoints/dashboard.py:671-679`
  - `pending_approvals`: `backend/app/api/v1/endpoints/dashboard.py:681-686`
  - `control_coverage`: `backend/app/api/v1/endpoints/dashboard.py:708-723`
  - `orphaned_items`: `backend/app/api/v1/endpoints/dashboard.py:738-743`
  - `kri_health`: `backend/app/api/v1/endpoints/dashboard.py:745-754`
  - `overdue_kris`: `backend/app/api/v1/endpoints/dashboard.py:755-761`
  - `risks_without_kri`: `backend/app/api/v1/endpoints/dashboard.py:771-778`
- **Impact:** The dashboard can show “no change” between quarters for metrics that actually did change, undermining the committee’s confidence in the numbers and creating audit-report risk.
- **Suggested fix:** Either (A) make all metrics truly period-based, or (B) explicitly label these as “current snapshot” metrics and remove them from the quarter-over-quarter delta calculation. If true as-of-quarter-end snapshots are required, store periodic snapshots or compute from historical/event data (Activity Log, status history, etc.).

### 4) Medium — “Closed risks in period” semantics are likely incorrect (uses `updated_at` + current `status == closed`)
- **Evidence:** `closed_risks` counts risks where `updated_at` is in the window AND `status == closed`: `backend/app/api/v1/endpoints/dashboard.py:646-652`
- **Impact:** This measures “currently closed risks that had any update in the period”, not “risks that transitioned to closed during the period”. It can overcount (non-status updates) and undercount (closed then reopened).
- **Suggested fix:** Track and count status transitions (e.g., via Activity Log `changes.status` or an explicit status-history table), or rename the metric to match what is actually computed.

### 5) Medium — Quarterly comparison endpoint is query-heavy (~34 DB queries/request) and uses filters that likely lack supporting indexes
- **Evidence:**
  - `get_quarter_metrics` issues many separate `db.scalar(select(func.count(...)))` calls: `backend/app/api/v1/endpoints/dashboard.py:637-779`
  - It executes twice per request (this quarter + last quarter): `backend/app/api/v1/endpoints/dashboard.py:801-802`
  - Some queried columns are not indexed in models:
    - `ControlExecution.executed_at` has no index: `backend/app/models/control_execution.py:28-40`
    - Orphaned item “unresolved” uses `resolved_at IS NULL` (no index), despite `status` being indexed: `backend/app/api/v1/endpoints/dashboard.py:739-742`, `backend/app/models/orphaned_item.py:49-51`
- **Impact:** Committee views can become slow as data grows (Activity Log and executions tables are expected to grow quickly), and may add avoidable load to the DB.
- **Suggested fix:** Consolidate counts where possible, add indexes aligned to the primary filters (e.g., `ControlExecution.executed_at`, `OrphanedItem.status` usage, composite indexes for common windows), and consider caching results for a short TTL for committee endpoints.

### 6) Low — Approval status filtering uses string literals (enum drift risk)
- **Evidence:**
  - Status filter hard-codes `["pending", "pending_privileged"]` and casts status to string: `backend/app/api/v1/endpoints/dashboard.py:681-686`
  - `ApprovalStatus` is imported but not used: `backend/app/api/v1/endpoints/dashboard.py:623-624`
- **Impact:** If enum values change (or DB stores different casing), metrics silently break or miscount.
- **Suggested fix:** Use `ApprovalStatus.PENDING.value` / `ApprovalStatus.PENDING_PRIVILEGED.value` (or compare against the Enum-typed column directly without casting).

## Frontend Findings

### 1) Medium — Committee UI gating is access-scope based, not permission-based (potential drift from backend auth model)
- **Evidence:** `isPrivileged = user?.access_scope === 'global'`: `frontend/src/pages/DashboardPage.tsx:75`
- **Impact:** If “committee eligibility” ever diverges from access scope (e.g., a role with a dedicated committee permission but not global scope), UI and backend can get out of sync.
- **Suggested fix:** Gate committee visibility using the same mechanism as the backend (recommended: a dedicated permission), or formally document that “committee == global scope”.

### 2) Medium — Quarterly widget assumes full API contract; missing keys would throw at render time
- **Evidence:** Widget dereferences `change.direction` and `change.absolute` without guards: `frontend/src/components/dashboard/QuarterlyComparisonWidget.tsx:129-155`
- **Impact:** Any backend contract drift (missing a metric key, changes missing for a key) can crash the Risk Committee view.
- **Suggested fix:** Add defensive rendering (null-check `change` and values) and/or strongly type the API response rather than `any` (`frontend/src/services/dashboardApi.ts:64-70`).

### 3) Low — Dashboard continues polling overview endpoints even when “Risk Committee” view is active
- **Evidence:** Overview data refresh interval runs unconditionally: `frontend/src/pages/DashboardPage.tsx:103-107`
- **Impact:** Unnecessary background load while reviewing committee dashboards.
- **Suggested fix:** Pause overview polling when `activeView === 'committee'`, or split polling logic per view.

# Phase 12.1-02 Findings — Activity Log Backend Audit

## Scope
- **Model/Migrations:** `backend/app/models/activity_log.py`, `backend/alembic/versions/e8dfb5de63a6_add_activity_log_table.py` (+ subsequent `actor_id` nullability migration)
- **Schemas:** `backend/app/schemas/activity_log.py`
- **Endpoints:** `backend/app/api/v1/endpoints/activity_log.py`
- **Integrations reviewed:** `backend/app/api/v1/endpoints/risks.py`, `backend/app/api/v1/endpoints/controls.py`, `backend/app/api/v1/endpoints/kris.py`, `backend/app/api/v1/endpoints/users.py`, `backend/app/api/v1/endpoints/approvals.py`, `backend/app/api/deps.py`

## Summary
- **Critical:** 0
- **High:** 4
- **Medium:** 5
- **Low:** 1

## Model/Migration Findings

### 1) High — API response schema requires `actor_id` but the model allows `actor_id = NULL`
- **Evidence:**
  - Model allows null: `backend/app/models/activity_log.py:60-62`
  - Schema requires non-null int: `backend/app/schemas/activity_log.py:14-16`
  - `log_activity` can write anonymous entries: `backend/app/core/activity_logger.py:48-50`
  - Auth endpoint creates anonymous login-attempt logs (`actor=None`): `backend/app/api/v1/endpoints/auth.py:43-56`
  - List endpoint validates rows through schema: `backend/app/api/v1/endpoints/activity_log.py:88-93`
- **Impact:** Any ActivityLog row with `actor_id = NULL` can cause `GET /activity-log` to 500 during Pydantic validation, especially for privileged users (who are not department-filtered and will see non-scoped entries).
- **Suggested fix:** Make `ActivityLogRead.actor_id` optional (and confirm all consumers handle nulls). Optionally add `actor_email`/`actor_role` snapshots if “anonymous” is unacceptable for audit requirements.

### 2) Medium — “Enum” fields are stored as free-form strings with no DB constraints
- **Evidence:** `backend/app/models/activity_log.py:52-58` stores `entity_type` and `action` as `String` with no check constraints; migration mirrors this (`backend/alembic/versions/e8dfb5de63a6_add_activity_log_table.py:26-35`).
- **Impact:** Data integrity relies on app correctness; malformed values can enter via bugs or scripts and break filtering/UI assumptions.
- **Suggested fix:** Add DB check constraints (or use DB enums) for `entity_type` and `action` if you want hard guarantees. If flexibility is intentional, document it.

### 3) Medium — Search implementation will not scale (full-table ILIKE scans)
- **Evidence:** Search uses `ilike("%...%")` across multiple columns: `backend/app/api/v1/endpoints/activity_log.py:64-72`. There are no supporting indexes for substring search in the migration (`backend/alembic/versions/e8dfb5de63a6_add_activity_log_table.py:40-48`).
- **Impact:** Activity Log queries can degrade quickly as the table grows (compliance logs typically grow fast), leading to slow UI and timeouts.
- **Suggested fix:** Move to full-text search (Postgres `tsvector`) or add trigram indexes for the searched columns; consider bounding time range by default (e.g., last 30/90 days).

### 4) Medium — “Tamper resistance” is API-level only (no DB immutability guarantees)
- **Evidence:** Model claims immutability (`backend/app/models/activity_log.py:40-46`), but there is no DB-level enforcement and only read endpoints exist (`backend/app/api/v1/endpoints/activity_log.py:19-34`).
- **Impact:** Anyone with DB access can update/delete rows; compliance posture may be overstated if “tamper-resistant” is interpreted literally.
- **Suggested fix:** Clarify the claim (“append-only via API”) or implement DB protections (restricted DB role, triggers, audit table, cryptographic signing/hash chaining).

### 5) Low — Unbounded `description`/`changes` can bloat DB and audit logs
- **Evidence:** `description` is `Text` and `changes` is `JSON` with no size guardrails (`backend/app/models/activity_log.py:66-71`).
- **Impact:** Large updates (long descriptions, large JSON) can inflate DB size and SIEM ingestion volume.
- **Suggested fix:** Truncate large fields, cap change capture to an allowlist, and/or implement a “details omitted” strategy with a separate storage mechanism.

## Endpoint & Integration Findings

### 6) High — “Old vs new” change capture is incorrect for Risk/Control updates (old values are lost before logging)
- **Evidence:**
  - Risk update commits before logging (`backend/app/api/v1/endpoints/risks.py:427-441`) and builds `changes` using `getattr(risk, k, None)` after mutation (`backend/app/api/v1/endpoints/risks.py:439-440`).
  - Control update does the same (`backend/app/api/v1/endpoints/controls.py:397-412`).
- **Impact:** Activity Log diffs are unreliable (often “old == new”), undermining the primary audit purpose.
- **Suggested fix:** Capture `old_values` before applying updates (or use SQLAlchemy attribute history), and build `changes` from (old, new) explicitly.

### 7) High — Approval workflow events are not logged (request creation / approve / reject)
- **Evidence:**
  - Approvals endpoints import `log_activity` but do not emit entries (no usage beyond `backend/app/api/v1/endpoints/approvals.py:25-26`).
  - Risk/Control/KRI endpoints create approval requests (e.g., edit/delete flows) but do not log the creation of those approval requests to Activity Log before returning 202 (e.g., `backend/app/api/v1/endpoints/risks.py:405-416`, `backend/app/api/v1/endpoints/controls.py:376-388`, `backend/app/api/v1/endpoints/kris.py:364-374`).
- **Impact:** Governance-critical actions (requests, approvals, rejections) are missing from the system-wide audit trail.
- **Suggested fix:** Add `ActivityEntityType.APPROVAL` logging on approval request creation and resolution, including department scoping based on the underlying resource.

### 8) High — User management changes are not logged (including password changes)
- **Evidence:**
  - Users endpoints import `log_activity` (`backend/app/api/v1/endpoints/users.py:14-16`) but `create_user` and `update_user` do not emit Activity Log entries (`backend/app/api/v1/endpoints/users.py:64-121`, `backend/app/api/v1/endpoints/users.py:272-337`).
  - `update_user` supports password updates (`backend/app/api/v1/endpoints/users.py:316-323`).
- **Impact:** High-value identity events are absent from the audit trail. This is a major compliance and incident-response gap.
- **Suggested fix:** Emit Activity Log entries for user create/update/deactivate, and explicitly redact/omit password and `hashed_password` from `changes`.

### 9) Medium — KRI update logging uses a different `changes` shape than the UI expects
- **Evidence:** KRI update logs `changes=update_data` (new values only) rather than `{field: {old,new}}`: `backend/app/api/v1/endpoints/kris.py:398-409`.
- **Impact:** UI diff rendering becomes misleading (shows “none → none” for primitives) and the DB record lacks old values.
- **Suggested fix:** Standardize `changes` format across all integrations and update the frontend to handle the final contract.

### 10) Medium — Activity log insertion happens after the main entity commit; failures can surface as 5xx after the write already succeeded
- **Evidence:** Entity endpoints commit the main object, then log activity, then commit again (e.g., `backend/app/api/v1/endpoints/risks.py:274-289`, `backend/app/api/v1/endpoints/controls.py:199-214`, `backend/app/api/v1/endpoints/kris.py:263-279`).
- **Impact:** If ActivityLog commit fails (serialization, DB issues), the user can receive an error while the underlying entity change has already been persisted, increasing duplicate/consistency risks.
- **Suggested fix:** Either (A) make ActivityLog write part of the same transaction (single commit) or (B) treat activity logging as best-effort and never fail the business operation (but then document the compliance tradeoff).

### 11) Medium — Department scoping relies on `access_scope`, not role names from the Phase 12 description
- **Evidence:** Scoping uses `get_user_department_ids()` which keys off `AccessScope.GLOBAL` (`backend/app/core/permissions.py:34-57`), not role names.
- **Impact:** If a Risk Manager / CRO user is misconfigured with a non-global scope, they may be unintentionally restricted, diverging from the Phase 12 access-control intent.
- **Suggested fix:** Enforce expected `access_scope` for privileged roles at user provisioning time, or gate “privileged visibility” by role/permission rather than access_scope.

## Areas Reviewed With No Issues Found
- Activity Log endpoint is permission-protected via `require_permission("activity_log", "read")` (`backend/app/api/v1/endpoints/activity_log.py:19-23`).
- Department scoping is applied before user-provided filters, preventing scope widening via query params (`backend/app/api/v1/endpoints/activity_log.py:45-62`).
- Migration provides indexes supporting the core filtered list use-case (entity_type/entity_id/actor_id/department_id/action/created_at) (`backend/alembic/versions/e8dfb5de63a6_add_activity_log_table.py:40-48`).

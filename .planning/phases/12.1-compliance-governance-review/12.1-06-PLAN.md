<task type="checkpoint:decision" gate="blocking">
  <decision>Define the authoritative access model for “Risk Committee” (includes Department Heads) and whether Admin is a “business-privileged” user</decision>
  <context>
Current behavior is inconsistent:
- Backend committee endpoints are authenticated-only (`deps.get_current_user`) and therefore callable by any user.
- Frontend hides committee UI for non-global scope users.
- Admin is seeded as `*:*` but is routed to Admin Console and does not see business navigation.

We need one clear rule so backend + frontend + tests match.
  </context>
  <options>
    <option id="privileged-plus-dept-head">
      <name>Committee access == privileged (global) + department_head (scoped)</name>
      <pros>
- Matches the requested UX for Department Heads.
- Reuses existing department scoping (`get_user_department_ids`) for non-privileged committee users.
- Minimal permission plumbing: can be implemented via role name check or a small helper.
      </pros>
      <cons>
- Requires careful scoping for metrics without `department_id` (e.g., `ApprovalRequest`, `OrphanedItem`) via joins.
      </cons>
    </option>
    <option id="permission-based">
      <name>Committee access == explicit permission (e.g., `dashboard:committee_read`) + department scoping for non-privileged</name>
      <pros>
- Most explicit and future-proof; can grant committee access beyond dept heads later.
      </pros>
      <cons>
- Requires new permission creation, role mapping updates, and migration/seed alignment.
- Adds another axis to reason about (scope vs permission).
      </cons>
    </option>
    <option id="admin-console-only">
      <name>Admin is console-only (cannot access business views like Activity Log / Committee)</name>
      <pros>
- Strong separation of duties if intended.
      </pros>
      <cons>
- Requires removing/overriding admin’s `*:*` and adding explicit blocks; may surprise reviewers.
      </cons>
    </option>
  </options>
  <resume-signal>Select: `privileged-plus-dept-head`, `permission-based`, or `admin-console-only` (and specify whether Admin can/can’t see business views like Committee).</resume-signal>
</task>

<task type="auto">
  <name>Task 1: Backend — enforce committee access for privileged users + Department Heads (scoped)</name>
  <files>backend/app/api/deps.py, backend/app/api/v1/endpoints/dashboard.py</files>
  <action>
Implement backend-side authorization so `/api/v1/dashboard/quarterly-comparison` and `/api/v1/dashboard/committee-summary`:
- cannot be accessed by non-committee users, and
- return department-scoped data for Department Heads.

Implementation guide (choose based on the decision):
- If `privileged-plus-dept-head`:
  1) Add a dependency in `backend/app/api/deps.py` (e.g., `get_current_committee_user`) that wraps `get_current_user` and allows:
     - privileged users (global scope), OR
     - `role.name == "department_head"`
  2) Apply the dependency to both committee endpoints.
  3) Apply scoping inside both endpoints for non-privileged users:
     - Filter risks/controls/KRIs/activity logs by department
     - For tables lacking a `department_id` column (e.g., approvals/orphaned items), scope via joins
- If `permission-based`:
  1) Add a new permission in `backend/app/db/seed.py` (resource/action) and ensure roles intended for committee have it.
  2) Gate both endpoints using `Depends(require_permission(...))` in `backend/app/core/security.py`.
  3) Keep UI gating aligned (Task 3).

Important: do NOT rely on frontend gating; backend must be the source of truth.
  </action>

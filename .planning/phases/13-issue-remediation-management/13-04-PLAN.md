<task type="auto">
  <name>Task 2: Add contextual issue create endpoint</name>
  <files>backend/app/api/v1/endpoints/issues.py, backend/app/schemas/issue.py, backend/app/services/issue_workflow_service.py</files>
  <action>
Add endpoint: `POST /api/v1/issues/contextual`.

Request contract:
- `entity_type`: `risk|control|execution|kri|vendor`
- `entity_id`: integer
- `title`: required
- `description`: optional
- `severity`: optional (`low|medium|high|critical`)
- `due_at`: optional datetime
- `owner_user_id`: optional

Behavior:
1. Resolve source entity and enforce read/write scope parity with existing issue visibility rules.
2. Resolve `department_id` from source entity:
   - risk/control/execution/kri: standard entity department resolution
   - vendor: use `vendor.department_id` if present; otherwise fallback to vendor owner department; if still unresolved return `409` with actionable detail.
3. Create issue + draft remediation plan (existing behavior parity with `POST /issues`).
4. Auto-create corresponding `IssueLink` record using the contextual entity.
5. Set `source_type` by entity type mapping:
   - risk/vendor => `manual`
   - control/execution => `control_execution`
   - kri => `kri_breach`
6. Emit activity logs for create + link actions.

Error behavior:
- Out-of-scope or non-existent entity => non-leaky `404`.
- Invalid owner assignment vs scope => `403`.
- Unresolvable vendor department => `409`.
  </action>
  <verify>cd backend && pytest -q tests/api/v1/test_issues_api.py -k contextual</verify>
  <done>Contextual issue intake is a single backend call with deterministic linking and scoped department inference.</done>
</task>

<task type="auto">
  <name>Task 1: Implement workflow service and guarded status transitions</name>
  <files>backend/app/services/issue_workflow_service.py, backend/app/api/v1/endpoints/issues.py, backend/app/schemas/issue.py</files>
  <action>
Add a dedicated workflow service to centralize allowed transitions:
- Issue status transitions:
  `open -> triaged -> in_progress -> ready_for_validation -> closed`
  with explicit deny on illegal transitions.
- Remediation plan transitions:
  `draft -> active -> blocked -> completed`.
- Exception transitions:
  `requested -> approved -> revoked|expired`.

Add workflow endpoints:
- `POST /issues/{id}/assign`
- `POST /issues/{id}/start-remediation`
- `POST /issues/{id}/update-progress`
- `POST /issues/{id}/request-exception`
- `POST /issues/{id}/approve-exception`
- `POST /issues/{id}/close`

Rules:
- assignment requires owner + due date (writes `Issue.owner_user_id` and `Issue.due_at`; creates/updates remediation plan owner/target as needed)
- start-remediation requires issue status in (`open`, `triaged`) and moves to `in_progress` with remediation plan status `active`
- update-progress updates `IssueRemediationPlan.progress_percent` (0..100) and optionally status/blocker_reason
- close requires remediation plan status `completed` (or progress 100%) and a required `validation_note`
- exception request creates `IssueException(status=requested)` and does not auto-close the issue
- exception approval requires elevated permission (`issues:approve`) and must set `expires_at` (time-bounded exception)
- exception expiry rule (enforced by deadline service): mark exception `expired` when `expires_at < now` and re-open issue if it was suppressed by exception

All transitions must log `ActivityLog` entries with structured change payloads.

Implementation detail (developer guidance):
- Keep endpoint modules thin; implement workflow mutations in `IssueWorkflowService`:
  - `assign_issue(...)`
  - `start_remediation(...)`
  - `update_progress(...)`
  - `request_exception(...)`
  - `approve_exception(...)`
  - `close_issue(...)`
- Use `build_change_set(...)` for issue/remediation/exception objects before mutating.
- Return 409 on invalid transitions; return 404 for out-of-scope issue IDs.
  </action>
  <verify>cd backend && pytest -q tests/api/v1/test_issue_workflow.py</verify>
  <done>Workflow is deterministic and enforced server-side via service-level state machine.</done>
</task>

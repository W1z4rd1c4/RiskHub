<task type="auto">
  <name>Task 1: Add issue + remediation domain models and DB migrations</name>
  <files>backend/app/models/issue.py, backend/app/models/__init__.py, backend/alembic/versions/*_add_issue_management_tables.py, backend/alembic/versions/*_extend_activity_entity_type_for_issues.py</files>
  <action>
Create an issue-management model set.

Recommended tables/models:
1) `Issue` (core finding)
- `id` (PK)
- `title` (required, max length 255)
- `description` (optional long text)
- `severity` enum: `low|medium|high|critical`
- `status` enum: `open|triaged|in_progress|ready_for_validation|closed`
- `source_type` enum: `manual|control_execution|kri_breach|audit`
- `source_id` (optional int; e.g., execution_id when `source_type=control_execution`)
- `department_id` (FK -> departments.id, nullable=False; see invariants below)
- `owner_user_id` (FK -> users.id, nullable=True, indexed)
- `created_by_id` (FK -> users.id, nullable=True)
- timestamps:
  - `opened_at` (tz-aware; default now)
  - `due_at` (tz-aware, nullable, indexed)
  - `closed_at` (tz-aware, nullable)
  - `created_at`, `updated_at` (follow existing conventions)

2) `IssueLink` (entity linkage)
- `id` (PK)
- `issue_id` (FK -> issues.id, ondelete=CASCADE, indexed)
- `risk_id` (FK -> risks.id, nullable)
- `control_id` (FK -> controls.id, nullable)
- `execution_id` (FK -> control_executions.id, nullable)
- `kri_id` (FK -> key_risk_indicators.id, nullable)

Constraints and invariants:
- Exactly one of (risk_id, control_id, execution_id, kri_id) must be non-null (enforce via CHECK constraint).
- `Issue.department_id` is derived from links when links exist:
  - if link is risk_id -> `risk.department_id`
  - if link is control_id/execution_id -> `control.department_id`
  - if link is kri_id -> `kri.risk.department_id`
Enforce at application layer (and add defensive tests).

3) `IssueRemediationPlan` (separate table so `13-02` workflow is executable)
- `id` (PK)
- `issue_id` (FK -> issues.id, unique, ondelete=CASCADE)
- `status` enum: `draft|active|blocked|completed`
- `progress_percent` int (0..100, default 0)
- `owner_user_id` FK -> users.id (nullable)
- `target_date` (tz-aware datetime; nullable)
- `blocker_reason` text nullable
- `completion_notes` text nullable
- `completed_at` tz-aware nullable

4) `IssueException` (exception tracking)
- `id` (PK)
- `issue_id` (FK -> issues.id, ondelete=CASCADE)
- `status` enum: `requested|approved|revoked|expired`
- `reason` text required
- `requested_by_id` FK -> users.id
- `approved_by_id` FK -> users.id nullable
- `requested_at`, `approved_at` tz-aware nullable
- `expires_at` tz-aware nullable (required for approved; enforce in workflow)

Schema requirements:
- Enums use non-native SQLAlchemy enum style consistent with existing models (`native_enum=False`, `validate_strings=True`).
- Index for common list filters:
  - Issue: `(status, severity)`, `(department_id, status)`, `(owner_user_id, status)`, `(due_at, status)`
  - IssueRemediationPlan: `(status)`, `(owner_user_id)`
- Add `ActivityEntityType` members for issue entities:
  - `ISSUE`
  - `ISSUE_REMEDIATION`
  - `ISSUE_EXCEPTION`
and include a migration that safely extends the DB-side constraint/enum for activity entity type (follow existing vendor entity expansion patterns).

Update model exports in `backend/app/models/__init__.py`.

<task type="auto">
  <name>Task 2: Add issue schemas, RBAC contract, and scoped API endpoints</name>
  <files>backend/app/schemas/issue.py, backend/app/schemas/__init__.py, backend/app/db/rbac_seed_contract.py, backend/app/api/v1/endpoints/issues.py, backend/app/api/v1/router.py</files>

Add Pydantic schemas for:
- create/update/list/read payloads
- enum values and filter params
- compact linked-entity fields (risk/control/execution/kri references)
- remediation plan and exception read models (mutated in 13-02)

RBAC contract:
- Add `issues:read`, `issues:write`, `issues:approve` permissions (canonical).
  - Note: Phase 13 is ŌĆ£findings/issuesŌĆØ, but API is `/issues`; permission keys must match API/domain naming to avoid drift.
- Assign role mappings consistent with governance intent (CRO/RM full; department heads scoped write; audit/compliance read).
- Ensure `admin` remains platform-only (no business data); do not grant `issues:*` to `admin`.

Create `issues` endpoints (prefix `/issues`) with backend scoping:
- `GET /issues` (filters: status, severity, owner_user_id, department_id, overdue, linked_risk_id, linked_control_id)
- `POST /issues`
- `GET /issues/{id}`
- `PATCH /issues/{id}`
- `POST /issues/{id}/links` and `DELETE /issues/{id}/links/{link_id}`

API behavior:
- Enforce permission checks via `require_permission("issues", "...")` (new resource) and enforce scoping:
  - privileged users (GLOBAL scope) can see all issues
  - department-scoped users can see issues in their dept
  - cross-department ownership exceptions:
    - if the issue is linked to a risk/control the user can read via owner exceptions, allow read (use `can_read_risk_id` / `can_read_control_id` patterns as needed).
- Normalize inaccessible resources to non-leaky 404 for read endpoints (match existing patterns in `risks.py`/`controls.py`).
- Emit `log_activity(...)` for create/update/link/unlink actions.

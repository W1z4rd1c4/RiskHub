<task type="auto">
  <name>Task 1: Preflight - confirm dynamic Tailwind class interpolation exists</name>
  <files>frontend/src/pages/LoginPage.tsx</files>
  <action>
Confirm LoginPage uses dynamic Tailwind classes like:
- `bg-${color}-...`
- `border-${color}-...`
- `text-${color}-...`

Tailwind content scanning cannot reliably detect these, so production builds can lose styling.

Wrong code example (current; `frontend/src/pages/LoginPage.tsx` around `#L71`):
```tsx
<div
  className={`w-8 h-8 rounded-full bg-${account.color}-400/10 border border-${account.color}-400/20 flex items-center justify-center text-${account.color}-400 text-xs font-bold`}
>
```

Why it breaks:
- Tailwind’s scanner only sees literal class strings in source. Interpolated strings like `bg-${...}` are not included in the generated CSS, so prod build can render with missing styles.
  </action>
  <verify>Open `frontend/src/pages/LoginPage.tsx` and locate the dynamic `bg-${account.color}` usage.</verify>

Replace interpolation with a static mapping (similar to the existing hover mapping):

Step-by-step:
1. Enumerate allowed colors (already implicitly enumerated in `colorClasses`).
2. Create a new mapping that includes the badge classes:
   - `badgeBg`, `badgeBorder`, `badgeText` (or a combined `badgeClassName`).
3. Use the mapping instead of interpolated strings.
4. Run `npm run build` to ensure the classes are included in output CSS.

Fix example sketch:
```ts
const badgeClasses = {
  rose: "bg-rose-400/10 border-rose-400/20 text-rose-400",
  purple: "bg-purple-400/10 border-purple-400/20 text-purple-400",
  // ...and so on for each color
} as const;
```

Then:
```tsx
<div className={`w-8 h-8 rounded-full border flex ... ${badgeClasses[account.color]}`}>
```

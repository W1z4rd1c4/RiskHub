<task type="auto">
  <name>Task 2: Implement a shared raw/blob request path and update reportApi</name>
  <files>frontend/src/services/apiClient.ts, frontend/src/services/reportApi.ts</files>
  <action>
Refactor so report downloads reuse the same base URL logic and auth header behavior:

Preferred approach:
- Add a method to `apiClient` that can fetch a binary response (Blob) and return `{ blob, headers }`.
- Ensure it uses the same URL builder as JSON requests and includes `Authorization` header.
- Update `reportApi` to call this method instead of building its own URL via `VITE_API_URL`.

Avoid:
- Any construction that can result in `undefined/reports/...`.
- JSON parsing on binary responses.

Step-by-step implementation option A (recommended):
1. In `frontend/src/services/apiClient.ts`, add a new method like:
   - `getBlob(endpoint: string, options?: RequestOptions): Promise<{ blob: Blob; headers: Headers }>`
2. Reuse the exact same URL construction logic already in `request()` (do not duplicate it in a subtly different way).
3. Attach the same `Authorization` header behavior from `request()`.
4. In `frontend/src/services/reportApi.ts`, replace direct `fetch(...)` with `apiClient.getBlob(url)` (or similar).
5. Keep filename parsing from `Content-Disposition` using returned headers.

Fix example sketch (apiClient):
```ts
async getBlob(endpoint: string, options: RequestOptions = {}) {
  // build URL same as request()
  // set Authorization header same as request()
  const response = await fetch(url.toString(), { ...options, headers });
  if (!response.ok) throw new Error(...);
  return { blob: await response.blob(), headers: response.headers };
}
```

Fix example sketch (reportApi):
```ts
const { blob, headers } = await apiClient.getBlob(`/reports/risks/pdf${qs}`);
```

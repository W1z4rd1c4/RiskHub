Implement a read-only mode:
- Define a TS type matching `/users/lookup` response (e.g., `UserLookup` or `UserDirectoryEntry`).
- In non-privileged mode, render only fields that exist (name/email/department/role_name) and explicitly hide:
  - user status toggle
  - access edit modal
  - any admin-only actions
- Remove fake timestamps/access_scope in UI and remove `as any` casting.

Keep:
- Privileged access management mode unchanged (uses `/access/...`).

Step-by-step implementation guidance:
1. Define `UserLookup` in `frontend/src/types/user.ts` (or a dedicated `types/userLookup.ts`) matching backend `UserLookup` schema returned from `/users/lookup`.
2. Update `userApi.listVisibleUsers()` return type to `UserLookup[]` (no anonymous inline type).
3. In `UsersPage`, store fallback users in `UserLookup[]`, not `UserRead[]`.
4. Update `UsersTable` (or add a parallel `UserDirectoryTable`) so it can render:
   - access users (privileged/dept head) OR
   - directory users (read-only)
5. Gate actions:
   - Only render toggle/edit actions when the current mode is privileged access mode.
6. Remove “fake timestamp/scope” columns from read-only mode; show only real fields.

Fix example sketch (typed fallback state):
```ts
const [directoryUsers, setDirectoryUsers] = useState<UserLookup[]>([]);

setDirectoryUsers(await userApi.listVisibleUsers());
```

  <name>Task 1: Fail-closed webhook signature verification in production</name>
  <files>backend/app/api/v1/endpoints/directory.py, backend/app/core/config.py, docker-compose.prod.yml</files>
  <action>
Change webhook verification so production cannot silently run with signature checks disabled:

Backend:
- Detect production mode via `Settings.debug` (or introduce a clear `ENV` field if already used elsewhere).
- If `settings.debug` is false and `settings.webhook_secret` is empty:
  - Reject webhook requests (e.g., 500 on misconfig or 401) and log a clear error.
- Keep the “skip verification” behavior only in debug mode.

Deployment:
- Update `docker-compose.prod.yml` so `WEBHOOK_SECRET` is required (use `${WEBHOOK_SECRET:?WEBHOOK_SECRET is required}`) when the directory webhook endpoint is deployed.

Avoid:
- Allowing “empty secret means accept any payload” in production.

Wrong code example (current; `backend/app/api/v1/endpoints/directory.py` around `#L45`):
```py
if not secret:
    logger.warning("WEBHOOK_SECRET not configured - skipping signature verification (INSECURE)")
    return
```

Why this is dangerous:
- An attacker can forge webhook payloads if the secret is unset, potentially creating/updating users depending on sync logic.
- Production compose currently allows empty secret (`docker-compose.prod.yml` sets `WEBHOOK_SECRET=${WEBHOOK_SECRET:-}`), so this insecure path is reachable in production.

Fix example sketch (fail-closed in non-debug):
```py
settings = get_settings()
if not secret:
    if settings.debug:
        logger.warning("WEBHOOK_SECRET not configured - skipping signature verification (DEBUG ONLY)")
        return
    logger.error("WEBHOOK_SECRET missing in production; rejecting webhook")
    raise HTTPException(status_code=500, detail="Webhook secret not configured")
```

Fix example (compose): in `docker-compose.prod.yml` use:
```yaml
- WEBHOOK_SECRET=${WEBHOOK_SECRET:?WEBHOOK_SECRET is required}
```
  </action>
  <verify>Manual: with DEBUG=false and empty WEBHOOK_SECRET, webhook requests fail; with a secret set, valid signature succeeds and invalid signature 401s.</verify>
  <done>Webhook signature verification is enforced (or the system fails loudly) in production.</done>

<task type="auto">
  <name>Task 2: Make scheduler singleton-safe in multi-worker deployments</name>
  <files>backend/app/core/scheduler.py, backend/app/main.py</files>
  <action>
Ensure only one scheduler runs in production:

Choose one approach and document it in code:

Option A (preferred with Postgres): advisory lock
- On startup, attempt to acquire a Postgres advisory lock (fixed key).
- Only start APScheduler if lock is acquired.
- Release lock on shutdown.

Option B: explicit env toggle
- Only start scheduler when `ENABLE_SCHEDULER=true`.
- Default it to false in web workers; run a single scheduler worker separately.

Avoid:
- Starting the scheduler in every Uvicorn/Gunicorn worker process.

Evidence of “per-worker start” risk:
- `backend/app/main.py` calls `start_scheduler()` in lifespan startup (around `#L41`).
- `backend/app/core/scheduler.py` only checks `if not scheduler.running` (around `#L52`), which is process-local and does not prevent multiple processes from starting their own schedulers.

Wrong code example (current; `backend/app/core/scheduler.py` around `#L50`):
```py
def start_scheduler():
    if not scheduler.running:
        setup_scheduler()
        scheduler.start()
```

Fix example sketch (Option B: env toggle):
```py
def start_scheduler():
    if os.getenv("ENABLE_SCHEDULER", "false").lower() != "true":
        return
    ...
```

Fix example sketch (Option A: Postgres advisory lock):
- Acquire `pg_try_advisory_lock(<constant>)` using a DB connection at startup.
- Only start scheduler if lock acquired.
  </action>
  <verify>Run backend with multiple workers and confirm only one process logs “Background scheduler started”.</verify>
  <done>Duplicate scheduled executions no longer occur.</done>

<task type="auto">
  <name>Task 3: Harden rate limiting (trusted proxies + bounded memory)</name>
  <files>backend/app/middleware/security.py, backend/app/core/config.py</files>
  <action>
Fix two issues in rate limiting:

1) Trust `X-Forwarded-For` only from trusted proxies:
- Add `trusted_proxies` config (CIDRs or explicit IPs) and default it to empty.
- If `request.client.host` is not trusted, ignore XFF and use `request.client.host`.

2) Bound in-memory state:
- Add eviction (TTL) for `RateLimitMiddleware.state` entries so idle keys are removed.
- Add a max key cap and evict oldest keys if exceeded (to prevent memory blow-ups under scan traffic).

Avoid:
- Blindly trusting XFF from arbitrary clients.
- Infinite in-memory growth.

Wrong code example (current; `backend/app/middleware/security.py` around `#L129`):
```py
forwarded = request.headers.get("X-Forwarded-For", "")
if forwarded:
    return forwarded.split(",")[0].strip()
```

Why this is a bypass:
- A direct client can set `X-Forwarded-For: 1.2.3.4` and rotate the header value to evade per-IP limits unless you only trust it from a known proxy/load balancer.

Wrong code example (memory growth; `backend/app/middleware/security.py` around `#L126`):
```py
self.state: Dict[str, RateLimitState] = defaultdict(RateLimitState)
```
There is no TTL/eviction for keys, so repeated unique keys can grow memory without bound.

Fix example sketch (trusted proxies):
```py
def _get_client_ip(self, request: Request) -> str:
    peer = request.client.host if request.client else "unknown"
    if peer not in self.settings.trusted_proxies:
        return peer
    xff = request.headers.get("X-Forwarded-For", "")
    return xff.split(",")[0].strip() if xff else peer
```

Fix example sketch (bounded memory):
- Store `last_seen` per key and periodically evict keys older than N minutes.
- Enforce `MAX_KEYS` and evict oldest keys when exceeded.
  </action>
  <verify>Unit tests (recommended): spoofed XFF from untrusted host does not alter client_ip; state does not grow without bound under repeated unique keys.</verify>
  <done>Rate limiting is harder to bypass and safer under load.</done>
</task>

<task type="auto">
  <name>Task 4: Tighten CSP in production without breaking development</name>
  <files>backend/app/middleware/security.py, frontend/nginx.conf</files>
  <action>
Split CSP by environment:

Backend CSP:
- In production, remove `'unsafe-eval'` unless there is a proven runtime requirement.
- Restrict `connect-src` to expected origins (avoid `https://*`).

Frontend nginx CSP:
- Align with backend direction; avoid `https://*` for `connect-src` in production.
- Keep necessary `connect-src` targets (same-origin API proxy, backend service name only if required).

Avoid:
- Breaking dev tooling (allow wider CSP in debug mode).
- Leaving production CSP overly broad.

Wrong code examples (current):

Backend CSP includes `unsafe-eval` and allows all https origins for connect-src (`backend/app/middleware/security.py` around `#L82` and `#L86`):
```py
"script-src 'self' 'unsafe-inline' 'unsafe-eval'"
"connect-src 'self' http://localhost:* https://*"
```

Frontend nginx CSP allows broad connect-src (including `https://*`) and `unsafe-eval` (`frontend/nginx.conf` around `#L65`):
```nginx
script-src 'self' 'unsafe-inline' 'unsafe-eval';
connect-src 'self' http://localhost:* http://backend:* https://*;
```

Fix example sketch (prod):
- Remove `'unsafe-eval'` (React/Vite production builds typically do not need it).
- Restrict connect-src to:
  - `'self'`
  - the API origin(s) you actually call (often `'self'` only when proxying `/api`)

Fix example sketch (dev):
- Keep additional localhost targets needed for dev servers.

  <name>Task 1: Confirm the crash condition exists</name>
  <files>backend/app/models/approval_request.py</files>
  <action>
Validate that `ApprovalRequest.created_at` uses `datetime.now(UTC)` before `UTC` is imported. This is a real runtime error (NameError) at import time, not a lint issue.

Also confirm Alembic imports `app.models` in `backend/alembic/env.py`, which means migrations are affected by this import crash.

Steps:
1. Open `backend/app/models/approval_request.py`.
2. Find the `created_at` mapped_column default lambda.
3. Confirm `UTC` is referenced there.
4. Find where `UTC` is imported in that file (currently near the bottom).
5. Confirm the import is below the reference (this is the actual bug).
6. Open `backend/alembic/env.py` and confirm it imports `app.models` (directly or indirectly), meaning any import-time crash in models blocks migrations.

Wrong code example (present in repo; `backend/app/models/approval_request.py` around `#L3`, `#L90`, `#L110`):
```py
from datetime import datetime
...
created_at = mapped_column(
    DateTime(timezone=True),
    default=lambda: datetime.now(UTC),
    nullable=False,
)
...
from datetime import UTC
```

Why this breaks:
- Python resolves names at runtime. When the class body executes, `UTC` is not defined yet, so importing `app.models` can raise `NameError: name 'UTC' is not defined`.
- Alembic loads models in `backend/alembic/env.py`, so migrations can fail before they even start.

Fix example:
```py
from datetime import datetime, UTC
...
created_at = mapped_column(
    DateTime(timezone=True),
    default=lambda: datetime.now(UTC),
    nullable=False,
)
```

- Fetch `medium/high/critical` thresholds via `get_config_int` (cache is fine).
- Compute ranges consistently:
  - critical: [critical..25]
  - high: [high..critical-1]
  - medium: [medium..high-1]
  - low: [1..medium-1]
- Ensure “high risk count” uses the intended high threshold (10) unless explicitly defined as “critical”.
- Ensure approval gating uses `high_risk_min_net_score` (10) consistently.

Avoid:
- Mixing “high” and “critical” semantics between endpoints.

Step-by-step refactor suggestion:
1. Add a small helper in backend (either in `backend/app/models/global_config.py` or a new `backend/app/core/risk_thresholds.py`) that returns:
   - `medium`, `high`, `critical` as ints
2. In each endpoint:
   - Replace local hardcoded dict/ranges with values returned from the helper.
   - Compute ranges consistently from the three thresholds.
3. Replace report summary computations that hardcode 16 with `critical`.
4. Replace department metrics hardcode 12 with `high` (or with a distinct label if it’s intentionally different; if so, document it explicitly in code and UI).

Example helper sketch:
```py
from app.models.global_config import get_config_int, ConfigDefaults

async def get_risk_thresholds(db):
    medium = await get_config_int(db, "medium_risk_min_net_score", ConfigDefaults.MEDIUM_RISK_MIN_NET_SCORE)
    high = await get_config_int(db, "high_risk_min_net_score", ConfigDefaults.HIGH_RISK_MIN_NET_SCORE)
    critical = await get_config_int(db, "critical_risk_min_net_score", ConfigDefaults.CRITICAL_RISK_MIN_NET_SCORE)
    return medium, high, critical
```

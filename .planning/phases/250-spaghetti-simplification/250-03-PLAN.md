# Plan 250-03: Simplify `approvals.py` (extract approval execution)

## Objective
Refactor `backend/app/api/v1/endpoints/approvals.py` (~960 lines) by extracting the business logic in `approve_request()` into a dedicated service/module, while preserving:
- tiered approval semantics (`PENDING` → `PENDING_PRIVILEGED` → `APPROVED`)
- side effects (resource updates/archives, KRI historization)
- notifications + background tasks
- activity logs (approval-level + entity-level)

## Non-goals / Invariants (do not change)
- Do not change routes, request/response schemas, or HTTP status codes.
- Do not change supported resource types:
  - **Only** `risk`, `control`, `kri` exist as `ApprovalResourceType` (no `KRI_VALUE` resource type).
- Do not change action types: `ApprovalActionType.DELETE` and `ApprovalActionType.EDIT`.
- Do not change `pending_changes` payload shapes already produced by existing endpoints.
- Do not change which activity log entries are created (entity types + actions + descriptions).

## Current State (verified)
- **File**: `backend/app/api/v1/endpoints/approvals.py`
- `approve_request()` handles:
  - Authorization: primary approver vs privileged approver
  - State transitions + timestamp fields (`primary_approved_at`, `privileged_approved_at`, etc.)
  - Execution for:
    - DELETE approvals: archive Risk/Control/KRI (no hard delete)
    - EDIT approvals:
      - Risk/Control: apply `pending_changes` fields
      - KRI: three distinct flows based on `pending_changes` shape:
        1) History correction (`history_entry_id`, `old_value`, `new_value`, `period_end`, ...)
        2) Value submission (`current_value`, `period_end`, optional `recorded_at`)
        3) Generic edit (+ optional `current_value` recording)
  - Logging:
    - Entity-level logs (`RISK`, `CONTROL`, `KRI`, `KRI_VALUE`)
    - Approval-level `APPROVAL` `APPROVE` log with status old/new
  - Transaction semantics:
    - explicit `await db.flush()` before `commit()` to surface integrity errors
  - Notifications:
    - on `PENDING → PENDING_PRIVILEGED`: notify privileged approvers
    - on final approval: background task notifies requester

## Implementation Steps

### 1) Create a service module to own approval execution
- [CREATE] `backend/app/services/approval_execution_service.py` (name can be `approval_service.py` if preferred, but avoid collisions with endpoint file names)
- Implement a small, explicit API (no over-engineered handler registry):
  - `async def load_approval(db, approval_id) -> ApprovalRequest` (with required relationships)
  - `def assert_can_approve(approval, current_user) -> tuple[bool, bool]` (returns `is_privileged`, `is_primary_approver` or similar)
  - `def apply_status_transition(approval, *, current_user, resolution_notes, is_privileged, is_primary_approver) -> bool`
    - returns `should_apply_changes`
  - `async def apply_side_effects(db, approval, *, current_user) -> None`
    - DELETE: archive Risk/Control/KRI + entity activity logs
    - EDIT:
      - Risk/Control: apply pending changes + entity logs
      - KRI: keep the existing 3-branch logic intact
  - `async def log_approval_approve(db, approval, *, actor, previous_status) -> None`
  - Keep `_notify_requester_resolved_background` as a background-task function in the endpoint module (it needs an `AsyncEngine` and sessionmaker).

### 2) Reduce `approve_request()` endpoint to orchestration only
- [MODIFY] `backend/app/api/v1/endpoints/approvals.py`
  - `approve_request()` should:
    1) load approval via the service
    2) authorize
    3) transition status
    4) if `should_apply_changes`, call `apply_side_effects`
    5) flush + commit (preserve existing error handling)
    6) if escalated to `PENDING_PRIVILEGED`, notify approvers and commit
    7) reload approval (for response) and schedule background requester notification on final approval

### 3) Preserve and document `pending_changes` shapes (critical)
In the service code, keep explicit guards for KRI approvals:
- History correction path triggers only when `history_entry_id` exists.
- Value submission path triggers only when both `period_end` and `current_value` exist.
- Otherwise treat as generic edit (+ optional value recording).
Do not “normalize” these payloads in this refactor.

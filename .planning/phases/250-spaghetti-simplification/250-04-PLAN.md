# Plan 250-04: Consolidate safe CRUD/approval patterns in `kris.py` and `risks.py`

## Objective
Reduce duplication between:
- `backend/app/api/v1/endpoints/kris.py` (~953 lines)
- `backend/app/api/v1/endpoints/risks.py` (~884 lines)

‚Ä¶without introducing risky ‚Äúgeneric query helpers‚Äù that assume common fields (Risk and KRI archive semantics differ).

## Non-goals / Invariants (do not change)
- Do not change RBAC rules (cross-department access, reporting-owner logic, etc.).
- Do not change archive semantics:
  - Risk uses `status != archived`
  - KRI uses `is_archived == False` (and non-privileged users must not see archived history)
- Do not change approval semantics or payload shapes:
  - `ApprovalRequest.action_type` (`DELETE` / `EDIT`)
  - `ApprovalRequest.pending_changes` (dict payloads)
- Do not ‚Äústandardize‚Äù status checks unless tests already cover it; preserve existing `PENDING` vs `PENDING_PRIVILEGED` checks as-is.

## Current State (verified)
Both endpoint files repeatedly implement the same mechanical steps when creating approval requests:
1) query for existing pending approval(s)  
2) build `ApprovalRequest(...)` with `action_type` + `pending_changes` (for edits)  
3) `db.add()` ‚Üí `await db.flush()`  
4) `log_activity(... ActivityEntityType.APPROVAL, ActivityAction.CREATE ...)`  
5) `await db.commit()`  
6) `IntegrityError` handling for the partial unique index (`ux_approval_pending`) ‚Üí return 409

## Implementation Steps

### 1) Create a small helper for ‚Äúcreate approval + log + commit‚Äù (no new abstraction layers)
- [CREATE] `backend/app/core/approval_request_factory.py` (or extend `backend/app/core/approval_helpers.py` if you prefer keeping approval utilities together)
- Add **one** focused helper:
  - `async def create_approval_request_with_audit(db: AsyncSession, *, approval: ApprovalRequest, actor: User, department_id: int | None, on_duplicate_detail: str) -> ApprovalRequest`
  - Behavior:
    - Adds + flushes the `ApprovalRequest`
    - Logs the approval-level activity entry (`APPROVAL` / `CREATE`)
    - Commits
    - On `IntegrityError` that looks like the pending unique constraint, raise `HTTPException(409, ...)` with a caller-provided message
  - Keep it intentionally narrow: do **not** try to hide ‚Äúexisting pending approval query‚Äù logic, because each endpoint uses slightly different criteria and messages.

### 2) Apply the helper in `risks.py` and `kris.py` approval-creation call sites
- [MODIFY] `backend/app/api/v1/endpoints/risks.py`
  - Replace the repeated try/except/flush/log/commit blocks when creating:
    - risk EDIT approvals (sensitive fields / priority-risk edit)
    - risk DELETE approvals
  - Keep the ‚Äúexisting pending approval‚Äù queries and HTTP 400 messages unchanged.
  - Keep the JSON response payloads unchanged (still return 202 with the same keys).

- [MODIFY] `backend/app/api/v1/endpoints/kris.py`
  - Replace the repeated approval creation blocks for:
    - KRI UPDATE approvals (non-privileged edits)
    - KRI DELETE approvals (non-privileged delete request)
    - KRI value submission approvals (non-privileged record-value)
    - KRI history correction approvals
  - Do not change the `pending_changes` shapes (they are interpreted later during approval execution).

### 3) Avoid risky ‚Äúshared list query builders‚Äù
Do **not** create generic `build_scoped_list_query(...)` helpers here. The scoping logic is security-sensitive and differs across resources (risk vs KRI vs cross-department ownership access).

If you want extra cleanup, keep it local:
- Extract private helpers inside each endpoint file, e.g. `_apply_risk_department_scope(...)`, but only if it reduces duplication and keeps behavior identical.

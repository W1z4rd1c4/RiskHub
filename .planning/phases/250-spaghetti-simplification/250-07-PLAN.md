# Plan 250-07: Simplify `UsersPage.tsx` (filters + render loop)

## Objective
Refactor `frontend/src/pages/UsersPage.tsx` (~670 lines) to:
- separate filter state + filtering logic from rendering,
- reduce JSX nesting in the table section, and
- eliminate React key warnings in the mapped rows (use keyed fragments).

## Non-goals / Invariants (do not change)
- Do not change fetch behavior or access-mode behavior:
  - `canManageAccess` → `accessApi.listAccessUsers()`
  - department head → `accessApi.listDepartmentAccessUsers()`
  - otherwise → `userApi.listVisibleUsers()`
- Do not change which filters exist today:
  - `searchTerm`, `roleFilter`, `scopeFilter` (access mode), permission resource/action filters (access mode)
- Do not change permission semantics of the filters:
  - permission filter checks `effective_permissions` strings (`resource:action`)
- Do not change expand/collapse semantics, AccessEditModal behavior, or activate/deactivate flow.

## Current State (verified)
- `UsersPage.tsx` mixes:
  - data fetching + mode selection
  - filter state and filter predicates
  - large table rendering with “expanded detail rows”
- In access mode, rows are rendered with a `<> ... </>` fragment per user; the fragment is not keyed (React warning risk).

## Implementation Steps

### 1) Extract filter state + filtering into a hook
- [CREATE] `frontend/src/hooks/useUsersPageFilters.ts`
  - Inputs:
    - `accessUsers: AccessUserRead[]`
    - `fallbackUsers: UserRead[]`
    - `isAccessMode: boolean`
  - State it owns (mirrors existing state):
    - `searchTerm`, `roleFilter`, `scopeFilter`, `permResourceFilter`, `permActionFilter`
  - Outputs:
    - setters for each filter
    - `filteredAccessUsers` and `filteredFallbackUsers` computed with `useMemo`
    - `hasPermFilters` boolean and a `resetPermissionFilters()` helper
  - Keep the existing `hasPermission(perms, resource, action)` semantics identical.

### 2) Extract the filter bar UI (optional but recommended)
- [CREATE] `frontend/src/components/access/UsersFilterBar.tsx`
  - Render the existing search + select controls (reuse `ThemedSelect`).
  - Keep props explicit (filters + setters + `isAccessMode`).
  - Do not change styling or labels.

### 3) Extract the table rendering (recommended)
- [CREATE] `frontend/src/components/access/UsersTable.tsx`
  - Inputs:
    - `isAccessMode`
    - `users` (already filtered and correct type for the mode)
    - `expandedUserId`, `onToggleExpand`
    - `canManageAccess`, `canManageUsers` (or just the booleans needed for action buttons)
    - action callbacks: `onEditAccess`, `onToggleStatus`
  - In the mapping, replace `<>...</>` with `React.Fragment key={user.id}>...</React.Fragment>` so the pair `(main row + expanded row)` is keyed correctly.

### 4) Refactor `UsersPage.tsx` to orchestrate
- [MODIFY] `frontend/src/pages/UsersPage.tsx`
  - Keep fetching and modal state in the page.
  - Replace inline filter logic with `useUsersPageFilters`.
  - Replace large inline filter-bar + table markup with the extracted components.

## Verification
Automated:
- `cd frontend && npm run build`

Manual smoke:
- Privileged user view:
  - search, role, scope, and permission filters work
  - expand/collapse works for regular, CRO, and admin rows
  - edit access modal opens/saves
  - activate/deactivate confirm works
- Non-privileged view:
  - search + role filter still works on fallback list

## Rollback Strategy
- Commit before extracting components.
- If props become too complex, revert and extract only the hook first (least risky), then re-extract UI components.

## Success Criteria
- `UsersPage.tsx` is shorter and easier to scan.
- Filter logic is unit-testable/pure (inside hook) and unchanged.
- No React key warnings for mapped rows.

- **Scope**: Medium
- **Risk**: Low (UI refactor; easy to verify)

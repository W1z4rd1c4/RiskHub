# Plan 250-01: Simplify `riskhub.py` (schemas + CRO dependency)

## Objective
Reduce complexity in `backend/app/api/v1/endpoints/riskhub.py` (1509 lines) by:
1) extracting inline Pydantic schemas into `backend/app/schemas/riskhub.py`, and  
2) converting the repeated CRO gating into a real FastAPI dependency.

Primary goal: **preserve exact behavior** (routes, responses, RBAC), while making the file easier to maintain.

## Non-goals / Invariants (do not change)
- Do not change any route paths, HTTP methods, status codes, query params, or response shapes.
- Do not change CRO access semantics: `RoleType.cro_only_roles()` remains the source of truth.
- Do not change DB queries or ordering behavior as part of this plan (only move/extract code).
- Do not introduce new “generic base router/service” abstractions.

## Current State (verified)
- **File**: `backend/app/api/v1/endpoints/riskhub.py` (1509 lines)
- Inline schemas currently defined in the endpoint file:
  - `RiskTypeRead`, `RiskTypeCreate`, `RiskTypeUpdate`
  - `GlobalConfigRead`, `GlobalConfigUpdate`
  - `ApprovalScenarioRead`, `ApprovalScenarioUpdate`
  - `PublicRiskTypeRead`
  - `RoleHubRead`, `RoleHubCreate`, `RoleHubUpdate`, `PermissionHubRead`
  - `DepartmentHubRead`, `DepartmentHubCreate`, `DepartmentHubUpdate`
- Several schemas still use **Pydantic v1-style** `class Config: from_attributes = True` (repo uses `pydantic>=2.5.0`), which is a known source of warnings.
- CRO gate is implemented as `require_cro(current_user)` called manually in many endpoints.

## Implementation Steps

### 1) Extract schemas into `backend/app/schemas/riskhub.py`
- [CREATE] `backend/app/schemas/riskhub.py`
  - Move the schema classes listed above verbatim.
  - Convert Pydantic config to v2 style for models that rely on ORM parsing:
    - Replace:
      - `class Config: from_attributes = True`
    - With:
      - `from pydantic import ConfigDict`
      - `model_config = ConfigDict(from_attributes=True)`
  - Keep schema class names identical to avoid rippling import changes elsewhere.
- [MODIFY] `backend/app/api/v1/endpoints/riskhub.py`
  - Delete the inline schema class definitions.
  - Import schemas from `app.schemas.riskhub`.
  - Keep `response_model=...` exactly the same.

### 2) Turn CRO gating into a dependency (remove repeated `require_cro(...)` calls)
- [MODIFY] `backend/app/api/v1/endpoints/riskhub.py`
  - Keep the existing `require_cro(current_user: User) -> User` function (logic unchanged).
  - Add a wrapper dependency that *also* performs auth:
    - `def get_cro_user(current_user: User = Depends(get_current_user)) -> User: return require_cro(current_user)`
  - For every CRO-only endpoint:
    - Replace `current_user: User = Depends(get_current_user)` + `require_cro(current_user)` with:
      - `cro_user: User = Depends(get_cro_user)`
    - Keep variable naming consistent and do not change behavior (only move where the check is executed).

### 3) Optional (only if it clearly reduces duplication): extract tiny mapping helpers
If you see repeated “manual schema building” blocks (e.g., `RiskTypeRead(...)` with repeated `isoformat()` calls), you may:
- Add private helpers in `riskhub.py` like `_risk_type_to_read(model, risk_count)` to reduce noise,
- But do not introduce new service layers or shared generic mappers.

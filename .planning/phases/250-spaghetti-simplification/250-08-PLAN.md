# Plan 250-08: Simplify `permissions.py` (readability-only, no semantic changes)

## Objective
Refactor `backend/app/core/permissions.py` (~399 lines) to improve readability and structure **without changing any access rules**.

## Non-goals / Invariants (do not change)
- Do not change privileged detection semantics:
  - `is_privileged_user(user)` is based on `user.access_scope == AccessScope.GLOBAL` (not role names).
- Do not change any permission decisions or error messages.
- Do not change `has_sensitive_field_changes` semantics:
  - must distinguish “field not provided” vs “explicitly provided as null”
  - must keep the special-case rule for `is_priority` (only `True → False` requires approval)

## Current State (verified)
- `has_sensitive_field_changes()` currently uses a sentinel `NOT_PROVIDED = object()` to treat:
  - missing field in payload as “no change”
  - explicit `None` as “clear the field” (counts as a change)
- There is test coverage for this behavior:
  - `backend/tests/test_sensitive_fields.py`

## Implementation Steps

### 1) Re-organize file into clear sections (mechanical move only)
- [MODIFY] `backend/app/core/permissions.py`
  - Group into sections:
    - Department scoping helpers
    - Permission evaluation (`has_permission`, `get_effective_permissions`, etc.)
    - Approval/committee access helpers
    - Sensitive-field detection
    - KRI reporting-owner helpers
  - Keep function names and signatures unchanged unless the change is purely internal and call sites are updated.

### 2) Make `has_sensitive_field_changes()` easier to read without changing behavior
- [MODIFY] `backend/app/core/permissions.py`
  - Move the sentinel to module scope (e.g., `_NOT_PROVIDED = object()`) to avoid re-creating it per call.
  - Extract the `is_priority` rule into a tiny private helper if it reduces branching, but preserve:
    - only `True → False` counts as a sensitive change
    - `False → True` does **not** trigger approval in this function
  - Keep return type as `(bool, dict)` and preserve the dict shape exactly.

### 3) Type-hint tightening (safe)
- Add explicit types for inputs/outputs where it improves readability (e.g., `dict[str, object]` for `old_data/new_data`), but do not change runtime behavior.

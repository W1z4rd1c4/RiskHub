# Plan 252-01: Audit Log Redaction Hardening

## Objective

Replace the current narrow audit change exclusion logic with a safe-by-default redaction policy while preserving useful non-sensitive audit detail.

## Assumptions

- Audit evidence remains valuable for “who changed what,” but sensitive content must never reach persisted or emitted `changes`.
- The same sanitized object must be used for DB persistence and SIEM emission.

## Steps

1. Create `backend/app/core/activity_redaction.py`.
2. Move redaction policy out of `activity_logger.py`.
3. Implement allowlist-first and risky-name redaction.
4. Redact unknown and free-text fields by default.
5. Sanitize once after normalization and before truncation.
6. Prevent redacted field names from leaking through descriptions.
7. Add focused redaction tests and SIEM parity assertions.
8. Update docs that describe audit evidence/export semantics.

<objective>
Add first-class handling for HTTP 202 “approval created” responses for edit/update flows (not just delete/archive) across Risk, Control, and KRI update UIs.

Purpose: Users must never believe an edit was applied when it was actually queued for approval; the UI should clearly show “submitted for approval” and what to do next.
Output: Updated forms/modals that detect `ApprovalCreatedResponse` on update calls, show an in-app confirmation banner, and avoid misleading navigation; targeted unit tests to prevent regressions.
</objective>

  <files>frontend/src/types/approval.ts, frontend/src/lib/approvalUi.ts</files>
  <action>
Implement a small, reusable helper that standardizes how update/edit flows react to 202 responses:
- Use `isApprovalCreatedResponse()` as the sole detection mechanism.
- Provide a single function (or tiny wrapper type) that returns either:
  - `{ kind: "applied" }` for normal 200 responses, OR
  - `{ kind: "approval", approvalId, message }` for 202 approval responses.

Make it easy for forms to:
- Show a dismissible inline banner (“Submitted for approval (ID: X). Changes not applied yet.”).
- Optionally link to `/approvals` if the current user can resolve approvals (or at least provide a “Go to Approvals” link for everyone, since it’s visible in nav).

Avoid:
- Duplicating string formatting across components.
- `alert()`; keep UX consistent with Phase 154-04 banners.

  <done>There is a single reusable helper used by subsequent tasks; no TS errors.</done>

<task type="auto">
  <name>Task 2: Wire approval handling into RiskForm, ControlForm, and KRI update UIs</name>
  <files>
frontend/src/components/RiskForm.tsx,

frontend/src/components/KRIForm.tsx,
frontend/src/components/kri/KRIModal.tsx,
frontend/src/pages/RiskDetailPage.tsx
  </files>
  <action>
Update ALL edit/update entry points to handle 202 approvals:

1) `RiskForm` (edit mode):
- When `riskApi.updateRisk()` returns `ApprovalCreatedResponse`, show the banner and do NOT navigate away as if saved.
- Keep the user on the form so they can understand the status (or navigate but only after showing an explicit “queued” state—choose one and be consistent).

2) `ControlForm` (edit mode):
- Same pattern for `controlApi.updateControl()`.

3) KRIs:
- `KRIForm` (edit mode) uses `kriApi.updateKRI()` which can return 202; add identical banner behavior.
- `KRIModal` “edit existing KRI” path: it currently calls `onSave` and closes; ensure approval response is surfaced (either inside modal or via parent callback that shows a banner).
- `RiskDetailPage.handleSaveKRI()` currently ignores 202; handle it so users don’t think the KRI was updated immediately.

Implementation notes:
- Prefer an inline banner near the top of the form/modal, with an explicit approval id.
- Make banners dismissible, and store the approval id in state so it doesn’t get lost on re-render.
- Ensure the form submit button reflects the queued state (e.g., disabled after submission, or changes “Submitted”).

Avoid:
- Regressing existing delete/archive approval UX added in Phase 154-04.
- Returning to list/detail pages without any visible indication that the edit is pending.
  </action>
  <verify>
`cd frontend && npm run build`
`cd frontend && npm run test:run`
  </verify>
  <done>
Risk/Control/KRI edit flows all surface 202 as “submitted for approval” with approval id, and do not claim the edit applied immediately.
  </done>
</task>

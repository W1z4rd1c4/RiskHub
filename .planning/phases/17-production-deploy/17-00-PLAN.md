# Plan 17-00: Admin Console Robustness Fixes

## Objective

Fix the "Active Users (24h)" and "Active Sessions" metrics in Admin Console which always show 0 or incorrect data due to timezone mismatches between Python datetime calculations and PostgreSQL TIMESTAMPTZ columns.

## Deep Analysis

### Problem 1: Active Users Count Shows 0

**Location:** `backend/app/api/v1/endpoints/admin.py`, line 263

**Buggy Code:**
```python
yesterday = datetime.now(UTC).replace(tzinfo=None) - timedelta(hours=24)
```

**Issue:** The `.replace(tzinfo=None)` strips timezone info, creating a naive datetime. When compared against `ActivityLog.created_at` (which is `DateTime(timezone=True)` → PostgreSQL `TIMESTAMPTZ`), the comparison fails silently.

### Problem 2: Active Sessions List Empty/Incorrect

**Location:** `backend/app/api/v1/endpoints/admin.py`, line 354 (SAME pattern)

**Buggy Code:**
```python
yesterday = datetime.now(UTC).replace(tzinfo=None) - timedelta(hours=24)
```

**Issue:** Same problem. This affects line 376 where `User.last_active_at >= yesterday` is compared. `User.last_active_at` is also `DateTime(timezone=True)`.

### Why `.replace(tzinfo=None)` Was Added (And Why It's Wrong)

The comment says "SQLite compatibility (tests)". The developer believed that:
- SQLite doesn't support timezone-aware datetimes natively
- Stripping timezone makes it "compatible"

**This is INCORRECT.** Proof:
- `backend/app/api/v1/endpoints/activity_log.py` line 68 uses:
  ```python
  date_from = datetime.now(UTC) - timedelta(days=90)
  ```
  **WITHOUT stripping timezone**, and this works correctly in both tests (SQLite) and production (PostgreSQL).

**SQLAlchemy handles the conversion automatically** when the column is `DateTime(timezone=True)`:
- PostgreSQL: Passes timezone-aware datetime directly
- SQLite: Converts to ISO 8601 string for storage/comparison

---

## Solution

**Simply remove `.replace(tzinfo=None)`** from both locations. This is proven to work by the existing pattern in `activity_log.py`.

> [!WARNING]
> **Do NOT use `func.now() - timedelta(hours=24)`**
> SQLAlchemy does NOT support subtracting a Python `timedelta` from a SQL function expression (`func.now()`). This would cause a runtime error:
> ```
> TypeError: unsupported operand type(s) for -: 'Function' and 'timedelta'
> ```

## Tasks

### Task 1: Fix `get_system_stats()` Active Users Query

**File:** `backend/app/api/v1/endpoints/admin.py`
**Location:** Lines 260-268

**Current Code (BROKEN):**
```python
# Active users in last 24h (approximation based on activity logs)
from app.models.activity_log import ActivityLog
# Use naive datetime for SQLite compatibility (tests), PostgreSQL handles both
yesterday = datetime.now(UTC).replace(tzinfo=None) - timedelta(hours=24)
active_users_result = await db.execute(
    select(func.count(func.distinct(ActivityLog.actor_id)))
    .where(ActivityLog.created_at >= yesterday)
)
active_users_24h = active_users_result.scalar() or 0
```

**Fixed Code:**
```python
# Active users in last 24h (approximation based on activity logs)
from app.models.activity_log import ActivityLog
# Timezone-aware datetime works for both PostgreSQL and SQLite via SQLAlchemy
yesterday = datetime.now(UTC) - timedelta(hours=24)
active_users_result = await db.execute(
    select(func.count(func.distinct(ActivityLog.actor_id)))
    .where(ActivityLog.created_at >= yesterday)
)
active_users_24h = active_users_result.scalar() or 0
```

**Changes:**
1. Remove `.replace(tzinfo=None)` from line 263
2. Update comment to reflect correct behavior

# Plan: AD Deprovisioning Check

## Objective
Implement periodic check for deleted AD users and auto-deactivate them in RiskHub.

## Execution Context
- **Phase**: 19 Production Deployment & Enterprise Integration
- **Plan**: 19-07
- **Depends on**: 19-05, 19-06 (AD integration)

## Tasks

### 1. Deleted User Detection
- [ ] Add method to `graph_service.py`:
  - `get_user_status(oid)` - Check if user exists in AD
  - Handle 404 response (deleted user)
- [ ] Store last sync timestamp per user

### 2. Scheduled Deprovisioning Check
- [ ] Create `backend/app/services/ad_deprovision_service.py`:
  - Query all users with external_id (AD-imported)
  - Check each against AD
  - Flag deleted users for deactivation
- [ ] Add APScheduler job for daily check:
  - Run at configurable time (default: 2 AM)
  - Log results to audit log

### 3. User Deactivation Flow
- [ ] When AD user is deleted:
  - Set `is_active = False`
  - Log activity: "User deactivated (AD sync)"
  - Optionally notify admin
- [ ] Preserve user data (don't delete, just deactivate)
- [ ] Show deactivated users in Admin Console with reason

### 4. Manual Sync Trigger
- [ ] Add Admin Console action:
  - "Check AD Status" button for individual users
  - "Sync All AD Users" button for full check
- [ ] Show last sync timestamp in user list

### 5. Orphan Handling
- [ ] When user is deactivated, trigger orphan flagging:
  - Risks owned by user → orphaned
  - Controls owned by user → orphaned
  - Use existing orphan service

## Verification
- [ ] Scheduled job runs and logs activity
- [ ] Deleted AD users get deactivated in RiskHub
- [ ] Manual sync triggers work from Admin Console
- [ ] Orphan items flagged correctly

## Output
- `backend/app/services/ad_deprovision_service.py`
- Updated scheduler configuration
- Admin Console "Sync AD" functionality
- Updated user list with sync status

<objective>
Make Settings documentation faster to navigate and enforce strict audience split:
- platform admins see only admin documentation,
- non-admin users see only user documentation,
- each document bubble/card exposes tags for quick scanning and filtering.
</objective>

<tasks>

<task type="auto">
  <name>Task 1: Enforce strict backend documentation audience contract</name>
  <files>backend/app/api/v1/endpoints/admin/docs.py, backend/app/schemas/admin.py</files>
  <action>
Refactor `/api/v1/admin/docs` so it returns exactly one audience set:
- `admin` role -> admin docs only,
- all non-admin roles -> user docs only.

Extend response metadata to support UI navigation:
- `audience` (admin|user),
- `tags` (short topic labels derived from filename and/or heading map).

Add per-file locale fallback to English when a locale-specific file is missing (for example `user-cs/vendors.md` gap).
Keep deterministic ordering (getting-started first, then alpha by title).
  </action>
  <verify>Role-based leakage is impossible in API responses and metadata is stable for frontend filtering.</verify>
  <done>Backend docs API is explicit, role-safe, and navigation-ready.</done>
</task>

<task type="auto">
  <name>Task 2: Simplify Settings documentation UX and add tag-driven navigation</name>
  <files>frontend/src/services/adminApi.ts, frontend/src/components/settings/DocumentationSettings.tsx, frontend/src/pages/DocumentationPage.tsx, frontend/src/i18n/locales/en/settings.json, frontend/src/i18n/locales/cs/settings.json</files>
  <action>
Update frontend doc models and UI to consume `audience` + `tags`.

In Settings documentation tab:
- keep one clean library view + one detail view,
- add visible tag chips on each document bubble/card,
- add quick-filter chips (All + per-tag),
- show active audience label ("Admin Documentation" or "User Documentation"),
- keep layout usable on desktop and mobile.

Apply the same metadata rendering to `/admin/docs` so platform-admin docs and settings docs behave consistently.
  </action>
  <verify>Users can narrow documents by tag in one click and immediately understand doc audience.</verify>
  <done>Documentation discovery is simplified and fast without changing the docs authoring workflow.</done>
</task>

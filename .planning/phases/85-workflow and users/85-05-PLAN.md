# Phase 85-05: Owner-Based KRI/Control Permissions

## Objective

Implement owner-based access control for KRIs and Controls with tiered approval workflow.

**Key Changes:**
1. Add `kri:record` permission for KRI value recording
2. KRI/Control owners from the department can edit their assigned items
3. KRI owners can submit values → Risk Owner approves (+ privileged for priority risks)
4. Fallback: If no explicit owner, Risk's owner is used

## Design Decisions (Confirmed)

| Decision | Choice |
|----------|--------|
| Primary approver for multi-risk Controls | Owner of highest-priority linked Risk |
| Risk Owner visibility | See ALL pending approvals for their risks |
| Privileged bypass | CRO/Admin can approve directly, skipping Risk Owner |

> [!WARNING]
> Existing approval requests will still use old workflow. Consider migration or grandfather clause.

### Backend: KRI Value Recording Endpoint

#### [MODIFY] [kris.py](file:///Users/stefanlesnak/Antigravity/Risk%20App/backend/app/api/v1/endpoints/kris.py)

Update `record_kri_value` endpoint:

1. Change permission from `risks:write` to `kri:record`
2. Allow if user is:
   - Privileged (CRO/Admin/Risk Manager) → apply immediately
   - KRI's `reporting_owner_id` OR Risk's `owner_id` → create approval with Risk Owner as primary approver
   - Falls within department scope
3. For priority risks (`risk.is_priority`), set `requires_privileged_approval=True`

**New approval flow:**
```
KRI Owner submits value
    → ApprovalRequest created (primary_approver = risk.owner_id)
    → If is_priority: requires_privileged_approval = True
    
Risk Owner approves (POST /approvals/{id}/approve)
    → If requires_privileged_approval: status = PENDING_PRIVILEGED
    → Else: apply and status = APPROVED
    
Privileged user approves (if required)
    → Apply and status = APPROVED
```

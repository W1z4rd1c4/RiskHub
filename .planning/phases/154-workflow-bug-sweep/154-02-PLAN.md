Update the Control↔Risk endpoints in `backend/app/api/v1/endpoints/controls.py` to support the legitimate “control owner” cross-department workflow:

Endpoints to fix:
- `GET /controls/{control_id}/risks`
- `POST /controls/{control_id}/risks`
- `DELETE /controls/{control_id}/risks/{risk_id}`

Required behavior:
- A user who is the `control_owner` of the control can access these endpoints even if the control’s `department_id` is different from the user’s department.
- Linking/unlinking must still require access to BOTH sides (control + risk). For the risk side, allow access via the same rules used elsewhere (department access OR ownership-based access as defined in `DISCOVERY.md` decision).

Avoid:
- Granting a control owner the ability to link to risks they cannot otherwise access (this becomes an authorization escalation vector).

Implementation guidance:
- Mirror the “has_access” pattern used in `backend/app/api/v1/endpoints/risks.py` linking endpoints: compute risk access via helper(s) (risk owner / KRI reporting owner / control-owner-of-linked-control / dept), compute control access via control owner / dept, then allow only if both are true.

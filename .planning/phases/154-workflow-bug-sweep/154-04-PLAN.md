<task type="auto">
  <name>Task 1: Add first-class handling for 202 ApprovalCreatedResponse across create/edit/delete flows</name>
  <files>
frontend/src/types/approval.ts,
frontend/src/services/riskApi.ts,
frontend/src/services/controlApi.ts,
frontend/src/services/kriApi.ts,

frontend/src/components/ControlForm.tsx,
frontend/src/pages/RiskDetailPage.tsx,
frontend/src/pages/ControlDetailPage.tsx
  </files>
  <action>
Implement a consistent UI response to approval-queued actions:
- Use `isApprovalCreatedResponse` to detect 202 payloads.
- For create/edit flows (RiskForm/ControlForm): show an explicit “Submitted for approval” message and avoid implying the change is already applied.
- For delete/archive flows (RiskDetailPage/ControlDetailPage/KRI delete): if the server returns approval JSON, show the same “Submitted for approval” confirmation and do NOT present it as “archived/deleted immediately”.

Make API typing honest:
- Delete endpoints currently return `void` in TS, but backend can return JSON (202). Update the TS return types to `void | ApprovalCreatedResponse` (or a discriminated union) and handle both.

Avoid:
- Silent navigation away without explaining the approval state.
- Duplicating ad-hoc string checks; keep logic in one helper.
  </action>
  <verify>
`cd frontend && npm run build` (TypeScript) and manual smoke: trigger an approval-required edit/delete and confirm the UI message is correct.
  </verify>
  <done>
Any approval-queued action results in explicit user feedback with the approval id (or a link to Approvals if applicable).
  </done>
</task>

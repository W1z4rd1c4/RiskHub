# Phase 156.1 Context: Admin Role & RBAC Hardening

## Why This Phase Exists

Deep review of admin-role implementation found a set of connected authorization and contract issues that can cause:

1. Unauthorized user-access mutations by non-admin global users.
2. Admin log settings API payload mismatches between backend and frontend.
3. Drift between canonical RBAC seed contract and legacy scripts/tests/mocks.

The fixes are coupled and should be delivered as one hardening slice.

## Locked Decisions

These were selected during plan refinement and are treated as defaults for implementation:

- Access-management mutation policy: **Admin/CRO only** for all writable access-user fields (`role_id`, `department_id`, `manager_id`, `access_scope`).
- Legacy seed script behavior: **Align + Warn** (do not leave wildcard-only legacy behavior).
- Admin log-config API migration strategy: **Frontend update + backend compatibility shim**.

## Primary Evidence Pointers

- Access endpoint currently gates with global scope, but non-role fields are still mutable by non-admin privileged users:
  - `backend/app/api/v1/endpoints/access.py`
- Frontend expects 2-field log config contract:
  - `frontend/src/services/adminApi.ts`
  - `frontend/src/pages/AdminConsolePage.tsx`
- Backend enforces 4-field split app/audit contract:
  - `backend/app/schemas/admin.py`
  - `backend/app/api/v1/endpoints/admin.py`
- RBAC canonical contract vs legacy script drift:
  - `backend/app/db/rbac_seed_contract.py`
  - `backend/scripts/seed_roles_permissions.py`
- Test/mock drift indicating admin wildcard assumptions:
  - `backend/tests/conftest.py`
  - `frontend/src/test/mocks/handlers.ts`
  - `docs/TESTING.md`

## Non-Goals

- No redesign of role taxonomy.
- No broad auth architecture rewrite (JWT/session model).
- No AD integration changes.

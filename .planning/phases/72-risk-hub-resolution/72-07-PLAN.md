<objective>
Make approval creation and approval-notifications scenario-driven and consistent across all Risk Hub flows (risks, controls, KRIs, and the generic /approvals endpoint).

Purpose: Today approvals are created in multiple endpoints with diverging behavior (missing notifications in risks/KRIs, hardcoded role routing, and no `requires_approval` bypass). This plan centralizes approval creation and makes notifications derive from `approval_scenarios.approver_roles` (incl. `risk_owner`).
Output: A single reusable `ApprovalWorkflowService` is used by endpoints to (1) consult `ApprovalScenario.requires_approval`, (2) create ApprovalRequests with `scenario_key`, and (3) send notifications to scenario approvers. Regression tests confirm scenario_key + notifications across endpoints.
</objective>

    3) Create notifications:
       - Use existing `NotificationService.create_notification(...)` with `NotificationType.APPROVAL_PENDING`, `resource_type="approval"`, `resource_id=approval.id`.
       - Keep errors per-recipient non-fatal (current behavior).

    4) Update tests (`backend/tests/test_notification_service.py`) to cover:
       - Scenario-driven routing: create ApprovalScenario in test DB, set `approval.scenario_key`, call `notify_approvers`, assert only scenario roles receive notifications.
       - `risk_owner` resolution: scenario roles `["risk_owner"]` and `approval.primary_approver_id` set → that user receives notification.
       - Legacy fallback: approval.scenario_key is null → behaves like today.

<task type="auto">
  <name>Task 2: Implement ApprovalWorkflowService (scenario lookup + create-or-bypass)</name>
  <files>backend/app/services/approval_workflow_service.py</files>
  <action>
    Create `backend/app/services/approval_workflow_service.py` to centralize approval creation. Keep it small and explicit.

    Recommended API:
    - `async def get_scenario(db, scenario_key) -> ApprovalScenario | None`
    - `async def requires_approval(db, scenario_key, default=True) -> bool`
    - `async def create_approval(db, *, scenario_key, resource_type, resource_id, resource_name, action_type, requested_by_id, reason, pending_changes=None, primary_approver_id=None, requires_privileged_approval=False) -> ApprovalRequest`
    - `async def create_or_bypass(db, *, scenario_key, ...) -> ApprovalRequest | None`

    Behavior for `create_or_bypass`:
    - If scenario exists and `requires_approval == False` → return `None` (caller applies change directly).
    - Else create ApprovalRequest (with `scenario_key`), `await db.flush()` to get approval.id, then:
      - `await NotificationService.notify_approvers(db, approval)` (now scenario-aware)
      - `await db.commit()`
    - Return ApprovalRequest.

    Important: do NOT silently swallow missing scenario rows in prod.
    - If scenario lookup fails because the key is missing, log a warning and default to “requires approval” (safer).
  </action>
  <verify>`rg -n \"class ApprovalWorkflowService|def create_or_bypass\" backend/app/services/approval_workflow_service.py`</verify>
  <done>There is a single reusable service for approval creation + scenario requires_approval checks + notifications.</done>
</task>

      - `DELETE /api/v1/risks/{id}` (non-privileged) → `risk_delete`

      - `PATCH /api/v1/risks/{id}` (non-privileged):
        - If sensitive fields changed → `risk_edit_sensitive` (pending_changes from `has_sensitive_field_changes`)
        - Else if risk is “priority/high” and Risk Hub policy wants approval:
          - Use `await is_critical_risk_async(risk, db)` (config-driven) to decide “high risk”
          - If true and scenario `risk_edit_priority.requires_approval` is true → create approval with pending_changes for ALL update fields.

    - Controls:
      - `PATCH /api/v1/controls/{id}` approval path → `control_edit`
        - Keep existing primary approver logic:
          - `primary_approver_id = await get_primary_approver_for_control(db, control.id)`
        - Keep existing tiered logic:
          - `requires_privileged_approval = await check_control_requires_privileged_approval(db, control.id)`
      - `DELETE /api/v1/controls/{id}` (non-privileged) → `control_delete`

    - KRIs:
      - `PUT /api/v1/kris/{id}` (non-privileged) → `kri_edit`
      - `DELETE /api/v1/kris/{id}` (non-privileged) → `kri_delete`
      - `POST /api/v1/kris/{id}/values` (non-privileged) → `kri_value_submit`
        - Ensure ApprovalRequest fields for tiered flow are set:
          - `primary_approver_id = kri.risk.owner_id`
          - `requires_privileged_approval = kri.risk.is_priority`
        - Keep pending_changes structure compatible with approvals executor logic (`approvals.py` expects KRI value submissions to store `current_value`, `period_end`, `recorded_at`).

    Must-fix gaps (these are current bugs):
    - Risk delete + risk edit approvals currently create NO notifications → after wiring, notifications must exist.
    - KRI edit approvals currently create NO notifications → after wiring, notifications must exist.

  <name>Task 1: Add scenario-role enforcement helpers in approvals endpoint</name>
  <files>backend/app/api/v1/endpoints/approvals.py</files>
  <action>
    Implement small, testable helpers near the top of `approvals.py`:

    - `async def _load_scenario(db, scenario_key) -> ApprovalScenario | None`
    - `def _scenario_roles(scenario) -> set[str]` (lowercase)
    - `def _is_role_allowed(role_name: str | None, roles: set[str]) -> bool`

    Then define the enforcement rules for scenario-backed approvals:
    - If `approval.scenario_key` is set AND scenario exists:
      - “Risk owner approval” is allowed only if:
        - `"risk_owner"` is in scenario roles AND
        - `approval.primary_approver_id == current_user.id`
      - “Privileged approval/rejection” is allowed only if:
        - `can_resolve_approvals(current_user)` is true AND
        - `current_user.role.name` is in scenario roles
      - For `pending_privileged` stage:
        - Only privileged approvers are allowed (risk_owner already approved).

    Backward compatibility (important):
    - If `scenario_key` is null OR scenario lookup fails → fall back to the current behavior (primary approver + can_resolve_approvals).
      This keeps legacy approvals resolvable and avoids bricking old data.
  </action>
  <verify>`rg -n \"_load_scenario|scenario_key\" backend/app/api/v1/endpoints/approvals.py`</verify>
  <done>Approval resolution code has clear helpers for scenario role enforcement with a safe legacy fallback.</done>

<task type="auto">
  <name>Task 2: Enforce approver_roles on approve transitions (PENDING + PENDING_PRIVILEGED)</name>
  <files>backend/app/api/v1/endpoints/approvals.py</files>
  <action>
    Update the approve endpoint authorization checks:

    Current behavior summary:
    - PENDING: primary approver OR privileged user can approve
    - PENDING_PRIVILEGED: privileged user can approve

    Replace the “privileged user” concept with scenario-aware privileged permission:
    - If scenario-backed:
      - PENDING:
        - Allow risk_owner only if role includes `risk_owner` AND user is `primary_approver_id`.
        - Allow privileged only if scenario roles include the user’s role name AND `can_resolve_approvals(user)` is true.
      - PENDING_PRIVILEGED:
        - Allow privileged only if scenario roles include role name AND `can_resolve_approvals(user)` is true.
    - If legacy (no scenario_key): keep existing behavior.

    Do not change the state machine:
    - Primary approver approval sets `primary_approved_at`
    - If `requires_privileged_approval` → status becomes `pending_privileged`
    - Else → status becomes `approved` and auto-executes changes
  </action>
  <verify>`pytest backend/tests -k \"approve.*scenario|pending_privileged\"` (after Task 4 adds tests)</verify>
  <done>Approve endpoint enforces scenario roles without breaking legacy approvals.</done>
</task>

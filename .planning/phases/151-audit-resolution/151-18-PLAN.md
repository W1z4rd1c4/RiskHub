# Plan 151-18: Production Security Guardrails & Concurrency Verification

## Objective
Implement final production guardrails for authentication and verify all concurrency/integrity fixes with a stress test.

## Execution Context
- **Files**: `backend/app/core/security.py`, `backend/app/main.py`, `backend/tests/stress/`

## Context
1. `MOCK_AUTH_ENABLED` is a convenience that must never be active in production.
2. We need a "master verification" that the ID generation and approval constraints actually work under load.

## Tasks

### 1. Hardening `MOCK_AUTH_ENABLED`
- [ ] Update `backend/app/core/security.py`.
- [ ] Add an environment check: `os.getenv("ENV") != "production"` OR `os.getenv("DEBUG") != "true"`.
- [ ] If `MOCK_AUTH_ENABLED=true` is set in production-like environments, emit a CRITICAL warning during app startup and force it to `false`.

### 2. Startup Assertions
- [ ] Add a `check_production_safety` function in `backend/app/main.py`.
- [ ] Fail fast if any dangerous development flags are active in a non-dev environment.

### 3. Concurrency Stress Test
- [ ] Create `backend/tests/test_concurrency_stress.py`.
- [ ] Simulate 50 concurrent risk creations.
- [ ] Simulate 20 concurrent duplicate approval requests for the same risk.
- [ ] Verify:
    - No duplicate Risk IDs.
    - No duplicate Pending Approvals.
    - No 500 errors (only 409s or 201s).

### 4. Audit Log Consistency Check
- [ ] Verify that every completed remediation step (ID generation retry, approval collision) emits a proper Audit Log entry if applicable (e.g., failed attempts don't necessarily need logs, but successful ones must have correct IDs).

## Verification
- [ ] Run the stress test suite: `pytest backend/tests/test_concurrency_stress.py`.
- [ ] Verify startup behavior locally by toggling `ENV=production`.

## Success Criteria
- [ ] System is resilient to simultaneous requests.
- [ ] Mock auth is effectively disabled for production deployment.
- [ ] No regression markers from the audit findings remain.

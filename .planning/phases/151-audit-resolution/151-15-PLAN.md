# Plan 151-15: Robust Risk ID Generation

## Objective
Replace the current race-condition prone Risk ID generation logic with a robust, atomic-retry pattern that handles concurrent creation requests without collisions.

## Execution Context
- **Files**: `backend/app/api/v1/endpoints/risks.py`
- **Dependencies**: Database unique constraint on `risk_id_code` (already exists).

## Context
The current implementation:
1. Composition of prefix from process name.
2. Attempt to lock latest row with `.with_for_update()`.
3. Count existing rows with prefix.
4. Compose `R{count+1}`.
5. Loop 100 times to check uniqueness in the app layer.

**Risk**: Concurrent transactions can read the same count, both find no collision in step 5, and then one fails on commit. The fallback for SQLite also makes the lock ineffective.

## Tasks

### 1. Refactor `create_risk` Endpoint
- [ ] Locate `create_risk` in `backend/app/api/v1/endpoints/risks.py`.
- [ ] Remove the complex `with_for_update` and `count` logic.
- [ ] Implement a generation function that:
    - Calculates the base prefix.
    - Gets the highest numeric suffix currently in use for that prefix using `func.max()` (numeric extraction if needed, or simple string max if padded).
    - Increments the suffix.
- [ ] Wrap the `db.commit()` in a retry block (e.g., up to 5 retries):
    - Catch `sqlalchemy.exc.IntegrityError`.
    - On collision, re-fetch the max suffix and increment again.
    - `await db.rollback()` before retrying.

<task type="checkpoint:decision" gate="blocking">
  <decision>Select remediation strategy for browser auth storage (toward Microsoft/Entra login)</decision>
  <context>
Current state: frontend stores `access_token` in `localStorage` and sends Bearer tokens. This increases impact of any XSS.

End goal (prod): users log in with Microsoft/Entra (Authenticator / M365-style).

Key architecture choice:
- SPA-with-MSAL keeps tokens in the browser (prefer in-memory, not localStorage), and the API accepts Bearer tokens.
- BFF (backend-for-frontend) uses OIDC Authorization Code + PKCE in the backend, and the browser only gets an httpOnly session cookie; tokens never touch JS.
  </context>
  <options>
    <option id="spa-msal">
      <name>SPA with MSAL (browser obtains tokens)</name>
      <pros>Standard for SPAs; enables direct Microsoft login; can keep backend stateless; minimal backend session changes</pros>
      <cons>Still a “tokens in browser” model; requires careful token storage (prefer memory), CSP allowlists for Microsoft endpoints, and strict XSS discipline</cons>
    </option>
    <option id="bff-oidc">
      <name>BFF (backend handles OIDC, browser uses httpOnly session cookie)</name>
      <pros>Best mitigation for token theft via XSS; simplest CSP/connect-src story for the SPA; aligns with “enterprise portal” deployments</pros>
      <cons>Larger backend change (sessions/cookies/CSRF); more moving parts; requires careful proxying and same-site cookie configuration</cons>
    </option>
    <option id="keep-current-dev-temporarily">
      <name>Keep current localStorage Bearer tokens temporarily (dev/demo only)</name>
      <pros>Zero breakage risk now; unblocks basic security testing on current stack</pros>
      <cons>Not acceptable as a final prod posture; must be replaced by SPA+MSAL or BFF before real deployments</cons>
    </option>
  </options>
  <resume-signal>Select: spa-msal, bff-oidc, or keep-current-dev-temporarily</resume-signal>
</task>

<task type="checkpoint:decision" gate="blocking">
  <decision>Select scope for timezone consistency</decision>
  <context>
Models currently mix timezone-aware and naive datetimes. Changing DB schema may be risky; normalizing at API boundaries can be safer.
  </context>
  <options>
    <option id="api-normalize-only">
      <name>Normalize datetimes at API boundary only</name>
      <pros>Low risk; avoids DB migrations; reduces client-side ambiguity</pros>
      <cons>DB still stores mixed representations; internal comparisons still need care</cons>
    </option>
    <option id="db-migrate-timezone-aware">
      <name>Standardize DB columns to timezone-aware</name>
      <pros>Strongest long-term consistency; fewer “naive vs aware” footguns</pros>
      <cons>Requires migrations and careful compatibility with SQLite-based tests and existing data</cons>
    </option>
  </options>
  <resume-signal>Select: api-normalize-only or db-migrate-timezone-aware</resume-signal>
</task>

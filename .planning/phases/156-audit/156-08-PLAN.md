<task type="checkpoint:decision" gate="blocking">
  <decision>Select remediation strategy for JWT storage</decision>
  <context>
JWT in `localStorage` increases impact of any XSS. Moving to httpOnly cookies is safer but can be breaking and requires CSRF considerations.
  </context>
  <options>
    <option id="keep-localstorage-harden">
      <name>Keep localStorage for MVP, harden mitigations</name>
      <pros>Low risk of breaking auth flows; minimal code churn; relies on CSP and general XSS hygiene</pros>
      <cons>Does not eliminate token theft if an XSS occurs</cons>
    </option>
    <option id="migrate-httponly-cookie">
      <name>Migrate to httpOnly cookie-based auth (recommended)</name>
      <pros>Mitigates token exfiltration via XSS; industry standard for browser apps</pros>
      <cons>Requires careful CORS/CSRF design; larger change across FE/BE; needs thorough testing</cons>
    </option>
  </options>
  <resume-signal>Select: keep-localstorage-harden or migrate-httponly-cookie</resume-signal>
</task>

<task type="checkpoint:decision" gate="blocking">
  <decision>Select scope for timezone consistency</decision>
  <context>
Models currently mix timezone-aware and naive datetimes. Changing DB schema may be risky; normalizing at API boundaries can be safer.
  </context>
  <options>
    <option id="api-normalize-only">
      <name>Normalize datetimes at API boundary only</name>
      <pros>Low risk; avoids DB migrations; reduces client-side ambiguity</pros>
      <cons>DB still stores mixed representations; internal comparisons still need care</cons>
    </option>
    <option id="db-migrate-timezone-aware">
      <name>Standardize DB columns to timezone-aware</name>
      <pros>Strongest long-term consistency; fewer “naive vs aware” footguns</pros>
      <cons>Requires migrations and careful compatibility with SQLite-based tests and existing data</cons>
    </option>
  </options>
  <resume-signal>Select: api-normalize-only or db-migrate-timezone-aware</resume-signal>
</task>

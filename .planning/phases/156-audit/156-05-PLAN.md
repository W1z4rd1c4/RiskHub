Backend rule (source of truth): `POST /api/v1/kris/{id}/values` allows only:
- users with `kri:submit`, OR
- KRI reporting owner (and potentially fallback owner semantics, if backend implements it).

Frontend changes:
1) Update `usePermissions().canRecordKRI`:
   - Remove permissive checks (`risks:write`, `approvals:write`) from `canRecordKRI`.
   - Keep it purely permission-based: `hasPermission('kri', 'submit')`.

2) In `KRIDetailPage.tsx`:
   - Compute a page-local `canRecordThisKri` based on:
     - `hasPermission('kri', 'submit')`, OR
     - `kri.reporting_owner_id === user?.id`
     - If there is a defined fallback in backend (risk owner when reporting_owner_id is null), mirror it by using `linkedRisk` once loaded.
   - Use `canRecordThisKri` (not global `canRecordKRI`) to show/hide the ‚ÄúRecord Value‚Äù button.

3) Update unit tests (`rbac_gating.test.tsx`):
   - Add/adjust assertions so ‚ÄúRecord Value‚Äù is:
     - Visible for `kri:submit`
     - Visible for reporting owner even without `kri:submit`
     - NOT visible for `approvals:write` alone
     - NOT visible for `risks:write` alone

Avoid:
- Letting privileged permissions (like `approvals:write`) imply KRI submission; backend explicitly separated these concerns.
  </action>

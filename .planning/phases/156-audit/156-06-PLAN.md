<task type="auto">
  <name>Task 1: Tighten CSP for the served UI (nginx) and make backend CSP environment-aware</name>
  <files>frontend/nginx.conf, backend/app/middleware/security.py, backend/app/core/config.py</files>
  <action>
Important note: In Docker deployment the SPA is served by nginx, so nginx CSP is what matters for browser security testing. Backend CSP covers API/docs responses and is still worth hardening, but it does not protect the SPA HTML if nginx serves it.

Part A (nginx CSP – primary for SPA):
- Update `frontend/nginx.conf` CSP header to remove `'unsafe-eval'`.
- Narrow `connect-src`:
  - Prefer `connect-src 'self'` as the default, because frontend calls `/api/...` on the same origin.
  - Avoid `https://*` and broad host wildcards.
- Keep existing allowances that are actually used:
  - `style-src` and `font-src` for Google Fonts (if still used).
  - `img-src` allowing `data:` and `blob:` as currently.

Part B (backend CSP – API responses):
Adjust `SecurityHeadersMiddleware` so CSP is strict in production and permissive only in debug (with an optional strict-testing toggle):

- In `settings.debug == True`:
  - Default behavior: keep current dev-friendly directives to avoid breaking local workflows.
  - Add an opt-in env var (example): `SECURITY_HEADERS_STRICT=true` that forces production-like CSP even in debug, so you can run basic security testing without switching off DEBUG.
- In `settings.debug == False`:
  - Remove `'unsafe-eval'` from `script-src`.
  - Narrow `connect-src` away from `https://*` where possible; default to `connect-src 'self'`.
  - If you truly need cross-origin API calls in production later, add an explicit allowlist env var (example: `CSP_CONNECT_SRC_EXTRA`) rather than broad `https://*`.

Avoid:
- Tightening in debug in a way that breaks local dev.
- Shipping `'unsafe-eval'` in production.
  </action>
  <verify>cd backend && pytest -q</verify>
  <done>nginx-served SPA and backend responses include CSP without `'unsafe-eval'` in strict/production mode; default dev remains functional.</done>
</task>

  <files>backend/app/core/logging.py</files>

Update logging configuration so file handlers and root logger respect `LOG_LEVEL`:

- Today, `app_handler`, `audit_handler`, and `root_logger` are forced to DEBUG.
- Change behavior:
  - Parse `log_level` once (already passed into `configure_logging`).
  - Set:
    - `root_logger.setLevel(parsed_level)`
    - `app_handler.setLevel(parsed_level)` (or a slightly more verbose level if explicitly desired)
    - `audit_handler.setLevel(parsed_level)` (or INFO minimum, depending on compliance requirements)

Constraints:
- Do not break existing admin log endpoints that read and parse logs.
- Keep JSON formatting unchanged.

Avoid:
- Making logs silent; INFO should remain visible.
- Inconsistent levels between console and file unless explicitly justified.
  </action>

# Phase 7.12: Permission Model Consistency & Null Department Handling

> **Source**: [PERMISSIONAUDIT.MD](file:///Users/stefanlesnak/Antigravity/Risk%20App/.planning/phases/07-user-management/PERMISSIONAUDIT.MD) - Medium Issues #1-4, Low Issue #1

## Objective

Address remaining permission model inconsistencies:
1. Null department items are globally accessible (security hole)
2. Department inheritance from manager not implemented
3. Approvals permission inconsistent between UI and backend
4. Permission logic duplicated between `AuthContext` and `usePermissions`

## Proposed Changes

### 1. Null Department Handling

#### [MODIFY] [permissions.py](file:///Users/stefanlesnak/Antigravity/Risk%20App/backend/app/core/permissions.py)

Change `check_department_access` to restrict null department items to privileged users:

```python
def check_department_access(item_dept_id: int | None, current_user: User) -> None:
    if item_dept_id is None:
        # Null department = only privileged users can access
        if not is_privileged_user(current_user):
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail="Access denied to unassigned items"
            )
        return
    # ... rest unchanged
```

### 2. Manager Department Inheritance

#### [MODIFY] [permissions.py](file:///Users/stefanlesnak/Antigravity/Risk%20App/backend/app/core/permissions.py)

Update `get_user_department_ids` to inherit from manager if user has no department:

```python
def get_user_department_ids(user: User) -> Optional[list[int]]:
    if can_see_all_departments(user):
        return None
    
    if user.department_id:
        return [user.department_id]
    
    # Inherit from manager if available
    if user.manager and user.manager.department_id:
        return [user.manager.department_id]
    
    return []  # No access
```

### 3. Approvals Permission Alignment

#### [MODIFY] [seed.py](file:///Users/stefanlesnak/Antigravity/Risk%20App/backend/app/db/seed.py)

Add `approvals:write` permission to Risk Manager role (already has via `can_resolve_approvals` but not explicitly).

#### [MODIFY] [ApprovalsPage.tsx](file:///Users/stefanlesnak/Antigravity/Risk%20App/frontend/src/pages/ApprovalsPage.tsx)

Use `can_resolve_approvals` capability from `/auth/me` response instead of `hasPermission("approvals", "write")`.

### 4. Permission Logic Centralization

#### [MODIFY] [usePermissions.ts](file:///Users/stefanlesnak/Antigravity/Risk%20App/frontend/src/hooks/usePermissions.ts)

Remove duplicate permission logic. Use only `AuthContext.hasPermission()` as the single source of truth.

## Purpose

RiskHub authorization is a business-policy contract, not only a set of route
guards. This document maps each major protected action to the backend guard,
service policy, response capability metadata, frontend gate, tests, and known
gaps that must stay aligned.

This is an audit and maintenance artifact. It does not introduce SQL RLS, route
changes, schema changes, or auth/session wire changes.

## Architecture Principles

- Backend enforcement is authoritative for protected data and mutations.
- Frontend gates mirror backend policy and must prefer backend `capabilities`
  metadata when available.
- Local frontend permission checks are compatibility fallbacks only. They must
  not become the sole authority for protected backend mutations.
- Protected business views default to least privilege and server-side scoping.
- Policy changes must update this file, the JSON mirror, business logic docs
  when user policy changes, and the testing guide when verification changes.
- Architecture-only refactors of authz-sensitive files must update this file and the JSON mirror, even when public policy is intentionally unchanged.
- Backend/frontend capability field-shape drift is validated by the compact
  [`capability-catalog.json`](./capability-catalog.json); it does not replace
  this contract matrix as the source of action semantics.

## Vocabulary

| Term | Meaning in RiskHub |
|---|---|
| Actor | The authenticated user and their role, permissions, scope, department, and ownership relationships. |
| Resource | The protected object or collection: risk, control, KRI, vendor, approval, report, dashboard metric, user, role, department, issue, or orphan. |
| Action | The operation being attempted: read, write, execute, approve, export, archive, restore, link, submit, resolve, or administer. |
| Backend guard | FastAPI dependency or endpoint-level authorization check, usually `require_permission`, `require_business_permission`, or a resource helper. |
| Service policy | Shared business authorization logic under `backend/app/core/_permissions/` or `backend/app/services/_*/`. |
| Privilege tier | Resolved per-approval boolean fivefold (`is_privileged`, `is_primary_approver`, `is_requester`, `scenario_match`, `privileged_scenario_match`) returned by `approval_scenario_policy.resolve_approval_privilege_tier`. |
| Privilege context | Per-request privilege bundle resolved by `backend/app/services/_approval_execution/privilege_context.py` and injected via `Depends(get_privilege_context)` on approval endpoints. |
| Response capability | Backend-computed action metadata returned to the UI, for example `can_update`, `can_archive`, or `can_request_correction`. |
| Frontend gate | UI visibility or route guard. It is a convenience layer and never replaces backend authorization. |
| Test evidence | Backend, frontend, or E2E tests that prove a contract row remains true. |

## Maintenance Rule

Update this contract and its JSON mirror whenever a diff changes any of these:

- endpoint guards or dependencies;
- service-level authorization policy;
- role, permission, scope, ownership, or approval authority behavior;
- response `capabilities` shape or semantics;
- frontend route gates, action gates, or capability fallbacks;
- report, export, dashboard, department, or vendor-linked visibility;
- tests that define the expected authorization behavior.

For architecture-only refactors, record the moved internal module and state whether user-visible policy changed.

If the business policy itself changes, update [`../BUSINESS_LOGIC.md`](../BUSINESS_LOGIC.md).
If the verification path changes, update [`../TESTING.md`](../TESTING.md).

Latest architecture-only note: Risk Hub config capability builders and
service-level permission checks were reconciled on 2026-05-06 by moving
role/department/risk-type/approval-scenario capability construction and
service perimeter checks into `backend/app/services/_authorization_capabilities/`.
On 2026-05-07, the global authorization projection was registered as the
Capabilities Module: `Capabilities.can(action, resource)` covers route/global
resource permissions, `MeCapabilities` is the flat session capability payload,
and per-row deny/restore/update metadata remains on the existing
`{Risk,Control,Vendor,Issue,KRI}Read.capabilities` schemas. Reserved endpoint
and permission surfaces are tracked in
`backend/app/api/v1/endpoints/_reserved_modules.toml`. User-visible
authorization policy and capability semantics are unchanged.
On 2026-05-09, KRI deadline reporting grace-window defaults moved to the
KRI period-algebra SSOT without changing notification visibility, manual
trigger authorization, or capability semantics.
On 2026-05-09, the frontend auth session provider was split into
`SessionContext.tsx`, `PreferencesContext.tsx`, and `AuthActionsContext.tsx`
behind the existing `AuthContext.tsx` compatibility shim. User-visible
authorization policy and capability semantics are unchanged.
On 2026-05-09, ownership-based visibility helpers moved behind
`backend/app/core/_permissions/_ownership_factory.py`, preserving the KRI
archived-record asymmetry: direct KRI owner checks include archived KRIs while
risk-scoped KRI owner checks filter archived KRIs out. Approval endpoints now
declare `Depends(get_privilege_context)` from
`backend/app/services/_approval_execution/privilege_context.py`; approval
authority semantics remain the existing `ApprovalPrivilegeTier` policy.
On 2026-05-09, the vendor status column was removed from backend schemas,
listing filters, report rows, and frontend vendor types after the forward-only
Postgres migration. Vendor archive visibility and report filtering now use
`is_archived`; authorization policy and capability semantics are unchanged.
On 2026-05-17, architecture-only deepening moved approved KRI value/history
execution, KRI vendor assignment reconciliation, archive side effects, register
listing planning, report export definitions, monitoring export-row projection,
dashboard issue metrics, access-user row projections, approval pending-change
presentation, and issue remediation capability projection behind their owning
Modules. User-visible authorization policy and capability semantics are
unchanged.

Strict capabilities rollout note: `STRICT_CAPABILITIES` is a frontend runtime
configuration flag and remains off by default until operational telemetry
qualifies the cutover. The `/auth/me` and `/auth/me/capabilities` handlers emit
structured `me_capabilities.parsed` and `me_capabilities.parse_error` log
counters. The flag may be enabled only after at least 99% successful
`me_capabilities.parsed` reads over a rolling seven-day window.

Latest behavior note: On 2026-05-07, explicit risk-list department filters were
made strict for department-scoped actors: ownership and linked-entity exceptions
still expand broad risk visibility when no explicit department is requested, but
`department_id=<id>` returns only risks in that department when the actor can
request it, and an empty list for out-of-scope departments. Sensitive Risk edits,
KRI edits, and KRI value submissions now share the privileged-approval rule:
privileged approval is required when the parent risk is priority or its net score
meets the configured high-risk threshold.

## Strict Cutover Policy

The frontend authorization layer was migrated from local permission predicates (`hasPermission(...)`) to backend-served capability projections (`MeCapabilities`). Strict-cutover is gated by a runtime feature flag.

### `STRICT_CAPABILITIES` flag
Source: `/auth/config` (mounted for clients at `/api/v1/auth/config`) runtime config; consumed in `frontend/src/services/capabilityFlags.ts:1-9` and read in `frontend/src/authz/policy.ts:90,100`. Defaults OFF until the flip-criterion is met.

When OFF: `policy.ts` calls `buildLegacyAuthz(...)` which retains `?? hasPermission(...)` fallbacks for resilience to missing `MeCapabilities` payloads.

When ON: `policy.ts` reads `meCapabilities.X` directly with no fallback; missing fields surface as immediate authorization failures.

### Flip criterion
Backend emits structured-log counters `me_capabilities.parsed` and `me_capabilities.parse_error` from the `/auth/me` handler. The flag may be flipped ON in production once `parsed / (parsed + parse_error) >= 0.99` over a rolling 7-day window.

### Invariant locks
- `tests/frontend/unit/src/authz/useAuthz.invariant.test.ts` — asserts no `?? hasPermission(` patterns inside `buildAuthz` body and no `hasPermission(` calls in `routing/business.tsx`.
- `tests/backend/pytest/architecture/test_w10_capabilities_all_allowlist_red.py` — asserts `__all__` of the capabilities module matches the curated allowlist TOML.

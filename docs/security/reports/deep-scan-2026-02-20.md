## Confirmed Findings (Prioritized)

### 1) HIGH — Refresh token replay/race allows duplicate valid sessions
- Impact:
Concurrent refresh requests with the same refresh cookie both succeed and mint separate valid refresh sessions. This breaks single-use rotation guarantees and allows persistent session continuation after token theft if attacker races refresh.
- Reproduction:
1. Obtain one refresh cookie via demo login.
2. Send two parallel `POST /api/v1/auth/refresh` requests with identical cookie.
3. Observe both return `200` and both emit distinct `Set-Cookie` refresh tokens.
4. Use both child refresh tokens; both can refresh again (`200`).
- Evidence:
`/Users/stefanlesnak/Antigravity/Risk App 2/tests/results/security/deep-scan-20260220-220251/manual-refresh-race-check.txt:8`
`/Users/stefanlesnak/Antigravity/Risk App 2/tests/results/security/deep-scan-20260220-220251/manual-refresh-race-check.txt:12`
`/Users/stefanlesnak/Antigravity/Risk App 2/tests/results/security/deep-scan-20260220-220251/manual-refresh-race-check-3.txt:7`
`/Users/stefanlesnak/Antigravity/Risk App 2/tests/results/security/deep-scan-20260220-220251/manual-refresh-race-check-3.txt:9`
- Code anchors:
`/Users/stefanlesnak/Antigravity/Risk App 2/backend/app/api/v1/endpoints/auth/refresh.py:42`
`/Users/stefanlesnak/Antigravity/Risk App 2/backend/app/api/v1/endpoints/auth/refresh.py:86`
`/Users/stefanlesnak/Antigravity/Risk App 2/backend/app/api/v1/endpoints/auth/_shared.py:79`
- Recommendation:
Use atomic, one-shot rotation semantics in DB.
Implement `SELECT ... FOR UPDATE` or optimistic `UPDATE ... WHERE revoked_at IS NULL` with rowcount check, then mint new session only if one row was successfully revoked.
Add a dedicated concurrent refresh regression test.
- Suggested owner: Backend Auth.
